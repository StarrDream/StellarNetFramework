## 1. 文档定位

本文档用于定义 **StellarNetFramework** 当前版本下的统一架构边界、模块职责、生命周期约束、协议规范、代码风格规范、目录组织规范、脚本粒度规范、房间路由规范、监听注册规范、房间录制与回放规范、断线重连规范、出厂自带业务规范以及后续框架扩展的统一接入标准。

本文档是后续架构设计、代码实现、联调测试、问题排查与代码评审的唯一统一依据。所有全局业务单元、房间业务单元、客户端基础设施、服务端基础设施、录制链路、回放链路、重连链路以及框架扩展，都必须服从本文档定义，不允许通过局部特判、口头约定或临时实现绕过统一边界。

本文档的目标不是增加理解负担，而是降低框架使用门槛。所有制度设计都必须以“让开发者第一天就能将精力投入游戏业务逻辑本身”为原则，减少命名猜测、流程猜测、接线猜测、目录猜测、脚本粒度猜测与生命周期猜测。

## 2. 框架定位与适用边界

### 2.1 定位宣言

**StellarNetFramework** 是一个面向 **200 个客户端连接规模** 的专用服务器、局域网或组网游戏（Dedicated Server / LAN / Virtual-LAN / P2P Co-op）联机二次开发框架。核心目标是**接管底层网络设施、生命周期管线与状态同步底座**，让开发者第一天就能将精力投入游戏业务逻辑本身。

本框架的适用边界必须被严格限定：

- 推荐使用类型：轻中量级多人合作游戏
- 允许场景：PVE 合作闯关、多人生存、联机建造、据点协作、休闲协作、副本合作、局域网联机玩法
- 严禁使用类型：高度竞技、强对抗、强公平性、低容错判定的高竞技 PVP 游戏
- 不推荐场景：强实时帧级对抗、重预测回滚、严苛命中一致性要求、职业电竞化目标产品

本框架的服务端定位是**规则权威**，不是面向高竞技对抗设计的强物理权威同步框架。  
因此它适合“可维护、可定制、可快速开发”的轻中量级多人合作项目，不适合追求极限对抗公平性的高竞技 PVP 产品。

在 **200 个客户端连接** 级别的专网场景下，本框架优先保证**开发效率、结构清晰度与可维护性**。框架仅提供基础服务，任何涉及具体业务的功能，例如多客户端竞争同一实体、物理校验、竞技对抗裁定等，均由开发者自行实现。

- 通过事件订阅监听发送出口来决定是否录制的黑盒 Replay 拦截机制

## 3. 核心设计目标

本框架的制度设计必须同时满足以下目标：

- 开发者易接入
- 协议边界清晰
- 生命周期清晰
- 路由隔离清晰
- 监听注册清晰
- 错误定位清晰
- 重连恢复边界清晰
- 录制与回放边界清晰
- 代码组织统一
- 目录层级统一
- 脚本粒度统一
- 出厂目录清单清晰
- 出厂骨架内容清晰
- Handle 注册风格清晰
- 协议元数据最小化
- 新人第一天即可快速进入业务开发

任何一项框架设计，如果要求开发者必须先理解大量隐式约定、隐藏规则或命名黑话后才能开始写业务代码，则视为不合格设计，应继续修订。

## 4. 核心架构原则

### 4.1 服务端绝对权威

房间真实状态、生命周期推进、结算判定、录制起止边界、断线重连恢复时机以及会话合法性，均由服务端统一裁定。客户端不具备房间业务权威，只负责：

- 协议事件收发
- 本地表现
- 本地结构重建
- 重连后的状态接入
- 回放模式下的本地重演控制

客户端不得自行确认以下事实：

- 房间是否已正式开始
- 房间是否已进入结算
- 当前业务状态是否成立
- 本地旧缓存是否可覆盖服务端当前状态
- 回放状态是否等价于在线真实房间状态

### 4.2 Shared / Server / Client 分层原则

框架必须严格遵守 Shared、Server、Client 三层边界：

- Shared 只放双端共同依赖的纯契约、协议基类、枚举、标识类型、注册元数据、序列化抽象等
- Server 只放服务端权威逻辑、房间管理、会话管理、消息路由、发送链、录制链、重连链等
- Client 只放客户端表现、入站分发、出站发送、会话上下文、本地回放控制等

### 4.3 协议事件驱动原则

所有跨网络数据流转必须通过显式协议事件完成，禁止依赖隐式状态同步或黑盒自动同步能力。

### 4.4 生命周期统一原则

全局业务单元、房间实例、房间业务单元、录制能力、回放控制能力都必须处于清晰、可控、可释放的生命周期中。禁止只创建不释放、只注册不解绑、只接入不退出。

### 4.5 易用性优先原则

代码风格、接入方式、命名规范、注册方式、生命周期入口、目录组织方式、脚本粒度方式都必须高度统一。统一不是为了形式美观，而是为了让开发者：

- 不需要猜该类放哪里
- 不需要猜相关代码分散在哪些目录
- 不需要猜协议怎么命名
- 不需要猜 `MessageId` 用哪个号段
- 不需要猜应该在哪里注册监听
- 不需要猜房间消息如何路由
- 不需要猜断线重连时谁负责恢复
- 不需要猜出厂自带哪些业务目录
- 不需要猜房间创建时哪些组件是强制挂载
- 不需要猜某个 Handle 的监听清单写在哪
- 不需要猜协议脚本应该拆成多少个文件

### 4.6 业务单元最小概念原则

对开发者开放的主开发口径中，服务端与客户端业务单元默认统一采用：

- Handle
- Model

其中：

- Handle 负责监听注册、请求处理、业务逻辑组织以及发送工具调用
- Model 负责纯数据状态

Sender 属于框架级发送工具，不作为业务单元固定层级强制存在。仅当发送逻辑本身形成高复杂度且具备稳定复用价值时，才允许额外抽出业务级 Sender。

### 4.7 同层级目录收口原则

同一个全局业务单元或同一个房间业务单元的相关代码，必须放在同层级目录下集中管理。不允许将同一业务边界下的 Handle、Model、局部事件、局部快照、局部配置、局部工具类型长期散落到多个无明显归属关系的目录中。

### 4.8 反过度碎片化原则

脚本文件组织必须服务于开发效率与理解效率，而不是服务于表面上的拆分数量。共享契约层协议、业务局部小类型、出厂骨架代码禁止无收益的过度拆分。

### 4.9 协议元数据最小化原则

协议类型必须坚持最小元数据表达。在四协议基类已经能够表达方向与域归属的前提下，不允许继续叠加重复语义的协议标注。协议层只保留业务真正需要的唯一标识元数据，并把扩展能力统一收口到协议基类预留字段中。

### 4.10 默认行为必须可预测

对发送链、回放链、事件链、销毁链、重连链等高风险流程，必须优先规定框架默认策略，避免实现方按习惯各自发挥。

### 4.11 禁止隐式上下文猜测

需要房间上下文、目标范围、组件清单、装配顺序等关键运行语义时，必须优先采用显式输入或显式元数据，不依赖隐式读取、动态猜测或反推。

### 4.12 统一装配优先于组件自注册

组件负责声明能力，不负责抢占装配控制权。Handle 绑定、组件接线与卸载清理由统一装配流程发起。

### 4.13 失败即阻断，拒绝半装配状态

组件未知、实例化失败、注册冲突、文件版本不匹配、关键上下文缺失时，必须立即阻断，严禁带病继续运行。

## 5. 域划分与作用域边界

### 5.1 全局域

全局域表示不依赖当前房间归属即可成立的业务范围，例如：

- 认证
- 建房 / 加房 / 离房请求入口
- 重连请求入口
- 公告系统
- 大厅全局聊天
- 商城
- 活动
- 邮件
- 回放下载与传输控制

### 5.2 房间域

房间域表示必须绑定有效房间上下文才能成立的业务范围，例如：

- 房间基础设置
- 房间拓展设置
- 准备状态
- 对局开始
- 局内操作
- 结算
- 房间内广播
- 房间业务单元间领域事件传播
- 房间全员事件广播旁路录制
- 房间聊天

### 5.3 回放域

回放域是客户端本地独立运行的观察与重演域。它消费录制结果或回放数据，不参与在线房间权威，不反向影响在线房间状态。

回放域的房间数据来源不是服务端实时网络下行，而是本地回放控制器根据回放文件内容，按录制顺序与时间线模拟服务端事件进行重演。

### 5.4 作用域边界

框架中的核心运行域至少包括：

- GlobalScope
- RoomScope

二者在 ServiceLocator、EventBus、生命周期清理和服务寻址边界上必须严格隔离，不允许退化成无作用域区分的大一统容器。

## 6. Shared / Server / Client 分层约束

### 6.1 Shared 层允许内容

Shared 共享的是**通信契约**，不是内部状态结构。允许放入 Shared 层的内容仅包括：

- 显式通信协议（`C2S_XXX` / `S2C_XXX`）
- 公共常量
- 双端共同成立的轻量值类型
- 协议基类与协议标记特性
- 不依赖单端运行时上下文的纯工具类或纯函数工具
- 底层统一传输封套等双端共同成立的数据载体
- 关键基础抽象接口定义，例如序列化接口、事件标记接口、服务标记接口、统一标识值类型
- 双端共同成立的协议注册元数据结构与注册表构建逻辑
- 双端共同成立的房间配置快照接口定义

### 6.2 Shared 层禁止内容

明确禁止放入 Shared 层的内容包括：

- 服务端全局模块内部结构
- 服务端房间业务组件内部运行态结构
- 服务端连接、会话、缓存等运行时对象
- 客户端表现层专用数据结构
- View、UI、PresentationModel、表现缓存结构
- 任何仅在单端成立的内部数据组织形式
- 客户端发送器实现
- 服务端发送器实现
- 任意依赖单端 Adapter 或单端运行时上下文的发送管线实现
- 任意具体第三方序列化实现
- 任意具体第三方网络库适配实现
- 服务端配置文件模型与服务端配置热重载实现
- 任意 `IRoomSettings` 的具体实现类

### 6.3 Shared 的核心定位总结

Shared 层共享的是**双端通信契约与共同成立的轻量工具**，不是“为了复用而复用”的跨端杂物仓库。

## 7. 统一标识值类型约束

框架中的关键运行时标识不得直接暴露为随意的原始类型，必须采用统一值类型语义。

约束如下：

1. `ConnectionId` 必须是框架级统一轻量值类型或等价统一包装类型
2. 底层网络库原生连接句柄必须在 Adapter 层映射为框架统一 `ConnectionId`
3. 上层业务、日志、路由、发送链与会话系统不得直接依赖底层网络库原生连接标识类型
4. `SessionId` 必须是服务端签发的会话标识，不等价于账号 ID、角色 ID 或客户端本地生成 ID
5. 客户端只能持有并回传 `SessionId`，不得自定义其权威值
6. `SessionId` 的失效、续期、替换与安全校验策略由服务端控制

该约束的目标是避免基础标识类型在不同实现者手中出现签名分裂。

## 8. 协议系统统一规范

### 8.1 四协议基类语义固定

协议方向与域归属必须由四协议基类直接推导：

- `C2SGlobalMessage`
- `C2SRoomMessage`
- `S2CGlobalMessage`
- `S2CRoomMessage`

这四个基类同时表达两层语义：

- 上下行方向
- 所属域边界

具体规则如下：

- 继承 `C2SGlobalMessage` 的协议，表示客户端发往服务端的全局域协议
- 继承 `C2SRoomMessage` 的协议，表示客户端发往服务端的房间域协议
- 继承 `S2CGlobalMessage` 的协议，表示服务端发往客户端的全局域协议
- 继承 `S2CRoomMessage` 的协议，表示服务端发往客户端的房间域协议

不允许通过额外配置、目录名、命名猜测、手工字段或并行标注重新定义协议方向与域归属。

### 8.2 四协议基类的制度化语义

1. `C2SGlobalMessage`
   - 客户端上行
   - 全局域
   - 不进入房间归属一致性校验链

2. `C2SRoomMessage`
   - 客户端上行
   - 房间域
   - 必须进入房间归属一致性校验链

3. `S2CGlobalMessage`
   - 服务端下行
   - 全局域
   - 不依赖当前房间运行时归属上下文

4. `S2CRoomMessage`
   - 服务端下行
   - 房间域
   - 必须保留可读取的房间运行时上下文

### 8.3 协议命名规范

所有业务协议命名必须严格遵守以下规则：

- 客户端上行协议只能命名为 `C2S_XXX`
- 服务端下行协议只能命名为 `S2C_XXX`

不允许出现以下命名风格：

- `ClientToServerXXX`
- `ServerToClientXXX`
- `ReqXXX` / `ResXXX`
- `RequestXXX` / `ResponseXXX`
- `CmdXXX` / `NotifyXXX`
- `MsgXXX`
- 不带 `C2S_` / `S2C_` 前缀的业务协议名

### 8.4 协议 ID 号段规范

协议 ID 号段必须采用以下统一规则：

- 框架出厂保留号段：`0 - 9999`
- 开发者自定义号段：`10000` 及以上

#### 框架号段用途

`0 - 9999` 只允许框架基础设施协议使用，例如：

- 认证基础协议
- 重连基础协议
- 房间骨架基础协议
- 框架保留的系统通知协议

业务开发者不得占用该号段。

#### 开发者号段用途

`10000` 及以上所有号段全部开放给开发者使用，用于：

- 游戏全局业务协议
- 游戏房间业务协议
- 玩法组件协议
- 扩展模块协议

框架不得继续侵占开发者号段。

### 8.5 协议只保留 MessageId 元数据

业务协议类型只保留 `MessageId` 作为协议唯一标识元数据。协议方向与域归属由四协议基类唯一决定，不再引入其他并行元数据重复表达相同语义。

### 8.6 明确禁止使用 DeliveryMode 协议标注

框架明文禁止在业务协议类型上使用以下一类标注：

`[DeliveryMode(DeliveryMode.ReliableOrdered)]`

以及其他等价用途的协议传输模式标注。

原因是：

- 四协议基类已经承担方向与域归属表达
- 协议层应坚持最小元数据原则
- 传输策略不应下沉为业务协议声明层的重复标注
- 该类标注会制造“协议还需要写额外元数据”的错误心智负担

### 8.7 四协议基类统一预留 ExtData 扩展字段

四协议基类统一预留以下扩展字段：

```csharp
public byte[] ExtData;
```

该字段用于承载业务层在特定场景下需要附加的自定义序列化数据。它不是替代常规协议字段设计的借口，不应成为无约束黑盒结构。

### 8.8 ExtData 的使用边界

ExtData 只应用于以下场景：

- 兼容性扩展承载
- 少量临时扩展数据透传
- 某些不值得提升为固定字段、但确有必要附带的数据段
- 需要由业务层自行序列化与反序列化的扩展负载

不应用于以下场景：

- 代替正常的强类型协议字段设计
- 把常规业务主数据全部塞入 ExtData
- 通过 ExtData 绕过协议评审与字段设计规范
- 把 ExtData 当作长期无约束黑盒结构

### 8.9 启动扫描校验要求

启动扫描阶段必须校验：

- 是否显式声明 `MessageId`
- 是否继承自四协议基类之一
- `MessageId` 是否重复
- `MessageId` 是否落在合法号段
- 是否存在非法 `DeliveryMode` 协议标注

任一校验失败，必须直接阻断启动。

### 8.10 MessageRegistry 是唯一协议元数据来源

`MessageRegistry` 只允许在启动装配阶段构建一次。运行期禁止动态覆盖、静默替换或平行维护另一套私有协议映射表。

### 8.11 非法方向协议必须阻断

- 服务端入站链收到 `S2C` 协议必须直接阻断
- 客户端入站链收到 `C2S` 协议必须直接阻断

### 8.12 禁止隐式同步

本框架禁止依赖 `SyncVar` 一类隐式自动同步机制。所有跨网络状态流转必须通过显式协议事件完成。

### 8.13 共享契约协议脚本聚合规范

共享契约层中的业务协议脚本必须按业务边界聚合，禁止无收益的一协议一文件式过度拆分。

推荐聚合粒度如下：

- 一个全局模块对应一个协议脚本
- 一个房间业务组件对应一个协议脚本

例如：

- `UserGlobalModuleMessages`
- `ReconnectGlobalModuleMessages`
- `RoomBaseSettingsRoomComponentMessages`
- `RoomChatRoomComponentMessages`

## 9. MessageRegistry 与协议注册约束

### 9.1 注册元数据要求

`MessageRegistry` 至少记录以下信息：

- `MessageId`
- 协议类型 `Type`
- 消息方向：`C2S` / `S2C`
- 消息域归属：`Global` / `Room`

其中：

1. `Direction` 与 `Domain` 必须由四协议基类直接推导
2. 不允许额外声明独立配置字段覆盖四协议基类已表达的方向与域归属
3. `MessageRegistry` 负责记录推导结果，不负责重新解释或重写协议类型语义

### 9.2 扫描边界

`MessageRegistry` 的启动扫描范围必须制度化：

1. 只扫描明确配置的程序集白名单
2. 只扫描带有 `[MessageId]` 的协议类型
3. 编辑器程序集、测试程序集默认不参与扫描
4. 注册表仅在启动阶段构建一次，运行期禁止动态覆盖或静默替换
5. 扫描到的协议类型必须且只能继承四协议基类之一
6. 若某协议类型未继承四协议基类，必须直接报错并阻断启动
7. 白名单程序集必须由客户端 / 服务端各自启动装配器显式指定
8. 白名单程序集配置不得通过 `NetConfig.json` 这类运行期配置文件驱动
9. 白名单程序集配置不得依赖程序集特性自动发现

### 9.3 发送链与接收链双向职责

`MessageRegistry` 不只是接收链解析表，也必须服务发送链。

强制约束如下：

1. 接收链通过 `MessageId` 查询协议类型与元数据
2. 发送链通过协议类型查询 `MessageId`
3. 同一份 `MessageRegistry` 元数据必须同时为收发两侧提供服务
4. 不允许发送侧再维护平行的类型到 ID 私有映射表
5. 客户端与服务端都允许注册“本端会发送但不会入站”的协议类型
6. `MessageRegistry` 的元数据结构、方向/域推导规则与注册表构建逻辑可放在 Shared 公共基础层
7. 客户端与服务端必须在各自启动阶段基于本端程序集白名单独立构建本端注册表实例
8. 双端共享的是注册规则与元数据结构，不共享运行时实例，不共享本端装配结果
9. 本端白名单程序集配置属于客户端 / 服务端各自启动装配职责，不属于 Shared 层静态常量职责
10. Shared 层中的注册表构建逻辑必须是显式纯构建器或等价纯工具形态，接受外部传入的程序集白名单并返回注册表实例
11. 禁止采用静态自驱动初始化、隐式全域扫描或运行期懒构建方式生成 `MessageRegistry`

### 9.4 重复 MessageId 阻断机制

若发现 `[MessageId]` 重复，框架必须：

- 输出 Fatal 级别日志
- 立即抛出异常
- 强制阻断服务端 / 客户端启动

## 10. 消息模型与底层传输封套

### 10.1 两层模型

框架采用“两层模型”表达网络消息：

1. 协议层四协议基类模型
2. 底层统一传输封套 `NetworkEnvelope`

二者职责必须分离。

### 10.2 协议层职责

协议层负责在类型系统层直接表达：

- 消息方向：`C2S` / `S2C`
- 消息域归属：`Global` / `Room`

### 10.3 传输层职责

`NetworkEnvelope` 只负责承载底层统一传输所需的公共运行时信息，不承担业务协议字段定义职责。

`NetworkEnvelope` 至少包含：

- `MessageId`
- `Payload`
- `RoomId`

### 10.4 RoomId 运行时上下文语义约束

1. `RoomId` 属于运行时房间上下文字段，不等价于开发者业务协议字段
2. 对 `C2SGlobalMessage` 与 `S2CGlobalMessage`，`RoomId` 默认允许为空、默认值或无效占位值
3. 对 `C2SRoomMessage` 与 `S2CRoomMessage`，`RoomId` 必须为有效值
4. 开发者定义的协议结构即使未显式声明 `RoomId` 字段，也不影响框架在 `NetworkEnvelope` 中强制保留房间上下文
5. 客户端与服务端分发链、日志、录像、状态过滤与回放隔离逻辑，均依赖该运行时 `RoomId`
6. `RoomId` 作为运行时上下文字段，不得被误解释为自动等价于协议体中的业务参数字段

### 10.5 ConnectionId 运行时注入约束

1. `ConnectionId` 不属于 `NetworkEnvelope` 字段定义
2. `ConnectionId` 属于服务端接收链的运行时上下文
3. 服务端在收到客户端消息后，必须从底层连接上下文中提取并附加有效 `ConnectionId`
4. 客户端业务逻辑不得持有或上传有效 `ConnectionId` 作为身份依据
5. 服务端发送客户端消息时，不向客户端暴露有效 `ConnectionId`

### 10.6 NetworkEnvelope 运行边界补充约束

1. Adapter 只负责字节流与 `NetworkEnvelope` 的封装 / 解封装
2. Adapter 不负责解释业务层 `RoomId` 参数语义
3. Adapter 不负责决定房间业务路由目标
4. 协议方向与域归属由四协议基类决定，而不是由包头字段猜测
5. `RoomId` 在框架层属于运行时上下文，不得上升为“所有业务协议都必须显式声明”的公共字段约束
6. 统一传输封套只有一个 `NetworkEnvelope`，不得再平行扩展第二套全局封套 / 房间封套定义，避免传输层模型分裂

## 11. 全局域与房间域协议职责边界

### 11.1 全局域协议

全局域协议包括：

- `C2SGlobalMessage`
- `S2CGlobalMessage`

其语义用于表达不依赖当前房间归属上下文的协议。典型场景包括：

- 登录
- 建房
- 加房请求
- 重连认证
- `C2S_Reconnect`
- 好友
- 公告
- 全局聊天
- 全局系统通知
- 回放下载与传输控制

约束如下：

1. 全局域协议默认不绑定当前房间归属
2. 当协议语义需要目标房间参数时，允许通过发送接口显式传入 `targetRoomId`
3. 该目标 `RoomId` 仅表示目标房间参数，不等价于当前归属房间凭证
4. 全局域协议不得进入房间归属一致性校验链
5. 全局域消息的 `targetRoomId` 不自动要求发送器执行“目标房间必须存在”的统一校验
6. 是否校验 `targetRoomId` 指向的房间真实存在，由具体发送场景与上层业务逻辑决定

### 11.2 房间域协议

房间域协议包括：

- `C2SRoomMessage`
- `S2CRoomMessage`

其语义用于表达房间域协议。典型场景包括：

- 房间业务组件请求
- 房间业务组件广播
- 局内公共状态变化
- 局内公共表现结果
- 房间内单体私有辅助消息

约束如下：

1. 房间域协议必须绑定有效的房间上下文
2. 房间上下文可不强制体现在开发者定义的协议字段中，但必须在框架运行时消息封套中存在
3. 房间域消息在服务端接收链中必须进入房间归属一致性校验链
4. 房间域消息在客户端接收链中必须保留可读取的房间上下文，用于状态过滤、回放隔离与延迟消息污染拦截

## 12. 发送链与发送器边界

### 12.1 发送接口分流语义

框架在发送侧必须按消息域提供显式分流接口语义：

- `NetSendGlobalMessage.Send(msg)`
- `NetSendGlobalMessage.Send(targetRoomId, msg)`
- `NetSendRoomMessage.Send(roomId, msg)`

约束如下：

1. 全局域消息默认不依赖房间上下文，可直接发送
2. 当全局域消息语义需要目标房间参数时，可通过带 `targetRoomId` 的重载发送
3. 房间域消息发送时必须显式提供 `roomId`
4. 发送接口必须在进入底层网络层前完成必要的参数校验与运行时上下文绑定
5. 发送接口分流属于调用层防误用措施，不替代协议语义本身
6. `NetSendRoomMessage.Send(roomId, msg)` 中的 `roomId`，必须最终绑定到 `NetworkEnvelope.RoomId`
7. `NetSendGlobalMessage.Send(targetRoomId, msg)` 中的 `targetRoomId` 仅表示该次发送语义中的目标房间参数，不自动将该消息提升为房间域消息

### 12.2 发送器实现归属边界与物理命名约束

`NetSendGlobalMessage` 与 `NetSendRoomMessage` 表达的是**发送分流语义**，不表示 Shared 层统一具体实现。

强制约束如下：

1. 客户端必须持有客户端本端发送器实现
2. 服务端必须持有服务端本端发送器实现
3. 双端不得共用同一具体发送器实现类型
4. Shared 层不得承载发送器实现，只承载协议契约、统一传输封套与纯工具
5. 客户端发送器只允许发送：
   - `C2SGlobalMessage`
   - `C2SRoomMessage`
6. 服务端发送器只允许发送：
   - `S2CGlobalMessage`
   - `S2CRoomMessage`
7. Shared 层不得定义跨端统一发送器接口，避免将单端发送管线抽象错误抬升为共享契约
8. `NetSendGlobalMessage` 与 `NetSendRoomMessage` 在文档中属于**发送分流语义名词**，不强制要求在 Shared 层存在同名物理类型
9. 工程落地时，推荐统一采用以下物理命名：
   - 客户端：`ClientGlobalMessageSender`、`ClientRoomMessageSender`
   - 服务端：`ServerGlobalMessageSender`、`ServerRoomMessageSender`
10. 客户端与服务端可在不破坏上述语义的前提下做项目级命名包装，但不得混淆全局域与房间域职责
11. 服务端发送器的对外方法应优先显式体现投递范围，不鼓励以一个模糊 `Send` 方法承载全部全局广播、全局单播、房间广播与房间单播语义

### 12.3 客户端房间域发送的 RoomId 绑定来源约束

客户端房间域发送必须采用**调用方显式传入 `roomId`** 的方式完成上下文绑定。

强制约束如下：

1. 客户端房间域发送必须使用 `NetSendRoomMessage.Send(roomId, msg)` 这类显式接口
2. 客户端发送器不得主动持有客户端房间上下文以隐式读取当前 `RoomId`
3. 调用方传入的 `roomId` 必须与当前有效房间上下文一致
4. 若当前不存在有效房间实例、当前状态不允许发送房间域消息，或传入 `roomId` 与当前房间上下文不一致，必须直接报错并阻断发送
5. 客户端房间域发送器只负责参数校验、协议序列化与 `NetworkEnvelope.RoomId` 绑定，不负责猜测当前房间上下文
6. 调用方可在进入发送器前做业务前置判断，但最终的安全阻断责任必须由客户端房间发送器自身承担
7. 任何仅依赖外层调用方保证合法性、发送器内部不再做最终校验的实现，都视为不符合框架约束

### 12.4 服务端发送目标解析边界

服务端消息投递范围由框架服务端发送层完成，Adapter 不认识房间。

强制约束如下：

1. Adapter 只面向连接级发送
2. Adapter 不提供“向整个房间发送”的房间语义能力
3. 房间成员展开、目标连接列表计算与广播范围判定，必须由服务端框架层完成
4. 发送器或其上层发送协调节点必须在进入 Adapter 前完成目标连接集解析
5. 广播目标集合为空时，不得视为一次有效公共广播发送
6. 框架基础投递分类只定义单播与全体广播
7. 排除发送者、排除某组成员等变体，属于目标集合计算策略，不额外上升为新的框架级发送语义分类

### 12.5 服务端发送协调节点与录制旁路拦截点

服务端发送链必须在“目标连接集合解析完成后、进入 Adapter 前”显式存在一个**服务端发送协调节点**或等价内部发送管线节点。

其职责边界必须严格限定如下：

1. 接收上层已确定协议类型的服务端发送请求
2. 根据协议域归属与投递范围完成目标连接集合解析
3. 完成 `NetworkEnvelope` 所需运行时上下文绑定
4. 在满足录制条件时主动触发 Replay 自动旁路写入
5. 将最终连接级发送任务交给 Adapter
6. 该节点只负责发送协调、投递范围判定与旁路拦截，不负责业务状态修改
7. 该节点不是 Adapter 的职责扩展，也不是房间业务组件的一部分
8. 该节点可作为内部实现结构存在，不强制要求暴露为独立公共模块名，但其职责边界必须被制度化固定
9. Replay 旁路拦截必须由发送协调节点主动发起，不得改为事件订阅监听方式
10. 发送协调节点应通过当前 `RoomInstance` 挂载的 `ReplayRecorder` 获取录制写入目标
11. 若当前房间未挂载有效 `ReplayRecorder`，则直接跳过录制分支，但不得影响正常发送流程

### 12.6 传输策略不由协议标注声明

网络传输策略必须由框架发送链、发送入口、底层通道或统一发送配置决定。业务协议类型本身不得再通过 `DeliveryMode` 一类标注声明传输策略。

## 13. 网络 Adapter 抽象接口形态

### 13.1 Adapter 抽象原则

框架必须通过统一网络 Adapter 抽象接口与底层网络库解耦。

其抽象形态必须满足以下约束：

1. 向上层暴露统一生命周期事件
2. 采用**事件委托上抛模式**
3. 不采用回调注入模式
4. 不采用继承重写作为主扩展方式
5. 上层通过订阅 Adapter 事件接收连接建立、断开与收包通知
6. Adapter 同时提供统一发送入口，供本端发送器调用

典型上抛事件语义至少包括：

- `OnConnected`
- `OnDisconnected`
- `OnDataReceived`

### 13.2 Adapter 发送入口职责边界

Adapter 的发送入口只负责底层传输，不负责房间业务语义决策。

强制约束如下：

1. Adapter 发送入口必须接收已完成上下文绑定的统一传输输入，例如 `NetworkEnvelope` 或等价传输对象
2. Adapter 不负责根据房间、会话或业务状态决定目标连接范围
3. Adapter 不负责从业务消息对象推导 `MessageId`
4. Adapter 不负责补全缺失的 `RoomId`
5. 发送器必须在调用 Adapter 前完成协议序列化、`MessageId` 解析、运行时上下文绑定

### 13.3 Adapter 职责红线

#### 允许职责

- 底层连接建立、断开、异常事件上抛
- 底层字节流与 `NetworkEnvelope` 的封装 / 解封装
- 从底层连接上下文提取并向服务端接收链提供 `ConnectionId`
- 在框架发送管线已完成上下文绑定后，透传 `RoomId`
- 调用上层统一入口，将数据转交给框架核心层

#### 禁止职责

- 禁止处理任何业务状态
- 禁止决定房间路由目标
- 禁止缓存房间内部数据
- 禁止进行玩法权限判断
- 禁止直接触发业务组件逻辑
- 禁止在 Adapter 层通过命名、字段内容或自定义规则重新判定协议方向与域归属

## 14. 统一网络入口、域路由器与 Handle 体系

### 14.1 总原则

框架必须采用：

- 统一网络入口
- 按域路由分发
- 按模块 Handle 处理
- 处理完成后再按需抛领域事件

约束如下：

1. 协议消息先进入统一网络入口
2. 统一网络入口只负责解析与转发，不负责业务处理
3. 域路由器只负责根据协议类型查找主处理者，不负责业务实现
4. 具体业务逻辑必须由模块 Handle 或房间业务单元 Handle 负责
5. 领域事件只能由主处理者在完成状态修改后按需抛出
6. EventBus 不直接承担原始协议广播职责

### 14.2 服务端统一入口

服务端必须存在统一的网络入口，例如 `ServerNetworkEntry`。

其职责仅包括：

1. 接收 Adapter 上抛的 `NetworkEnvelope`
2. 通过 `MessageRegistry` 查找协议类型
3. 完成反序列化
4. 判断协议属于 `C2SGlobalMessage` 还是 `C2SRoomMessage`
5. 将消息转发给：
   - `ServerGlobalMessageRouter`
   - 房间域一致性校验链入口

明确禁止：

- 在入口层编写登录、建房、战斗、准备等业务逻辑
- 在入口层直接修改业务状态
- 在入口层直接抛领域事件

### 14.3 客户端统一入口

客户端必须存在统一的网络入口，例如 `ClientNetworkEntry`。

其职责仅包括：

1. 接收服务端下发的 `NetworkEnvelope`
2. 通过 `MessageRegistry` 查找协议类型
3. 完成反序列化
4. 将协议先交给客户端状态协议过滤器
5. 协议通过过滤后，再判断其属于 `S2CGlobalMessage` 还是 `S2CRoomMessage`
6. 将消息转发给：
   - `ClientGlobalMessageRouter`
   - 当前在线 `ClientRoomInstance` 内部的 `ClientRoomMessageRouter`

明确禁止：

- 在入口层直接刷新 UI
- 在入口层直接创建房间内实体表现
- 在入口层直接处理项目级表现逻辑
- 在入口层承担本地回放消息注入职责

### 14.4 服务端全局请求监听位置

所有 `C2SGlobalMessage` 必须进入 `ServerGlobalMessageRouter`。

其职责包括：

1. 管理全局域协议到 Handle 的注册表
2. 按消息类型查找全局模块主处理者
3. 调用对应全局模块 Handle

### 14.5 服务端房间请求监听位置

所有 `C2SRoomMessage` 在通过房间归属一致性校验后，必须进入目标 `RoomInstance` 内部的 `ServerRoomMessageRouter`。

其职责包括：

1. 管理房间域协议到 Handle 的注册表
2. 按消息类型查找房间业务组件主处理者
3. 调用对应房间业务组件 Handle

### 14.6 客户端全局广播监听位置

所有 `S2CGlobalMessage` 必须进入 `ClientGlobalMessageRouter`。

其职责包括：

1. 管理客户端全局协议到 Handle 的注册表
2. 按消息类型查找客户端全局模块主处理者
3. 调用对应客户端全局模块 Handle

### 14.7 客户端房间广播监听位置

所有在线态下的 `S2CRoomMessage` 必须进入当前在线 `ClientRoomInstance` 内部的 `ClientRoomMessageRouter`。

其职责包括：

1. 管理客户端房间域协议到 Handle 的注册表
2. 按消息类型查找客户端房间业务组件主处理者
3. 调用对应客户端房间业务组件 Handle

回放态下，录制消息不得从在线网络入口进入，而必须由 `ReplayPlaybackController` 直接注入到本地回放房间实例的 `ClientRoomMessageRouter` 入口。

### 14.8 协议类型与领域事件类型禁止混用

网络协议类型与领域事件类型必须严格分层，不得混用。

强制约束如下：

1. 网络协议类型不得直接作为 EventBus 事件类型使用
2. 领域事件类型不得直接作为网络协议发送
3. 协议输入与领域事件传播必须使用不同类型体系
4. 协议基类与领域事件标记接口不得混合继承形成语义污染

### 14.9 View 与协议监听红线

View、UI、动画、音效、特效、镜头系统不得直接监听 `C2S_XXX` 或 `S2C_XXX` 协议。

其合法依赖来源只能是：

1. Model 变化
2. Service 回调
3. EventBus 中的领域事件
4. 开发者自定义的表现层上下文通知

## 15. 统一注册器与 Handle 注册规范

### 15.1 为什么需要 Registrar

应引入统一注册器抽象，而不是直接让业务 Handle 裸持 Router。Registrar 的目的，是让开发者接触到的是：

- 统一注册语义
- 统一链式风格
- 最少理解成本

### 15.2 注册器应满足的目标

- 支持链式 Register
- 支持泛型注册
- 隐藏 Router 内部实现
- 保证全局业务单元和房间业务单元风格统一
- 降低新人接入门槛

### 15.3 全局业务单元统一消息监听注册规范

全局业务单元必须采用：

- 一个 Handle 一个 `RegisterAll(...)`
- 在同一个入口中集中列出该 Handle 监听的所有消息
- 每条消息对应命名清晰的处理方法

所有全局业务单元消息注册，统一要求使用链式 Register 风格，例如：

`xxx.Register<S2C_XXX>(OnS2C_XXX).Register<S2C_YYY>(OnS2C_YYY);`

或服务端上行版本：

`xxx.Register<C2S_XXX>(OnC2S_XXX).Register<C2S_YYY>(OnC2S_YYY);`

同一个全局业务单元 Handle 的监听清单必须集中写在一个位置，不允许拆散到多个无明显边界的角落中。

### 15.4 房间业务单元统一消息监听注册规范

房间业务单元统一在房间实例驱动的初始化接入阶段集中完成消息注册。实际消息监听注册应由对应的 RoomHandle 统一承接。

所有房间消息处理方法统一使用：

- `OnC2S_XXX`

涉及恢复流程或服务端下行模拟注入场景时，也应保持显式命名。

房间业务单元内部的 RoomHandle 应采用链式 Register 风格，在集中注册入口中统一书写。

房间业务组件 Handle 只能注册属于当前业务边界的消息，不允许把多个房间业务边界混写到同一个 RoomHandle 的注册清单中。

注销必须与房间销毁阶段绑定，确保房间销毁后不残留旧监听。

## 16. 代码风格统一规范

### 16.1 类型命名规范

建议采用以下统一命名：

- 协议：`C2S_XXX` / `S2C_XXX`
- 全局业务处理入口：`XXXHandle`
- 全局业务数据模型：`XXXModel`
- 房间业务处理入口：`XXXRoomHandle`
- 房间业务数据模型：`XXXRoomModel`
- 全局模块协议聚合脚本：`XXXGlobalModuleMessages`
- 房间业务组件协议聚合脚本：`XXXRoomComponentMessages`
- 领域事件：`XXXEvent`
- 设置快照：`XXXRoomSettings`

### 16.2 业务单元结构规范

客户端与服务端业务单元默认统一采用：

- Handle
- Model

Handle 是主入口，负责：

- 协议监听注册
- 上行或下行消息处理
- 业务逻辑组织
- 调用框架级 Sender 工具发消息
- 协调必要的上下文依赖

Model 是纯状态层，负责：

- 保存业务状态
- 保存缓存数据
- 承载可恢复的运行时数据

### 16.3 协议处理方法命名规范

所有协议处理方法必须采用显式命名：

- `OnC2S_XXX`
- `OnS2C_XXX`

不推荐使用：

- `Handle`
- `OnMessage`
- `Process`
- `Do`
- `Callback`

### 16.4 日志规范

所有错误日志必须至少包含：

- 所在类名
- 当前对象名或房间标识
- 触发对象
- 关键状态变量

发现异常时必须：

- `Debug.LogError(...)`
- 立即 `return`
- 阻断当前逻辑继续向下执行

### 16.5 前置拦截规范

所有常规业务逻辑统一使用 Early Return。禁止深层嵌套式条件结构。

### 16.6 常规业务逻辑禁用 Try-Catch

常规业务逻辑中禁止使用 try-catch 掩盖错误。仅在以下场景允许：

- 外部 I/O
- 不可控第三方 SDK
- JSON 解析
- 框架级原子回滚保护

### 16.7 注释规范

注释必须全部使用中文专业注释。注释应说明：

- 为什么这么做
- 该逻辑在架构中的职责边界
- 该设计用于规避什么问题

### 16.8 统一日志字段建议

所有关键错误日志、阻断日志、状态迁移日志建议统一携带以下上下文字段中的必要子集：

- 类名
- 实例名或对象名
- `MessageId`
- `RoomId`
- `ConnectionId`
- `SessionId`
- 当前状态
- 关键变量值

## 17. 目录组织统一规范

### 17.1 同一业务边界代码必须同层级集中放置

同一个全局业务单元或同一个房间业务单元的相关代码，必须放在同层级目录下集中管理。

### 17.2 目录后缀只表达业务归属

全局业务目录推荐组织为：

- `UserGlobalModule/`
   - `UserGlobalModuleMessages`
   - `UserHandle`
   - `UserModel`

房间业务目录推荐组织为：

- `ReadyRoomComponent/`
   - `ReadyRoomComponentMessages`
   - `ReadyRoomHandle`
   - `ReadyRoomModel`

其中：

- `GlobalModule` 只是全局业务目录后缀
- `RoomComponent` 只是房间业务目录后缀

### 17.3 目录组织目的

进入某个业务目录时，应能完整看到：

- 该业务的协议在哪
- 该业务的入口在哪
- 该业务的状态在哪
- 该业务的局部依赖类型在哪

### 17.4 目录命名允许历史兼容

出于历史兼容原因，目录层级允许保留：

- `XXXGlobalModule`
- `XXXRoomComponent`

但目录内的主业务类型，必须优先统一为：

- `XXXHandle`
- `XXXModel`
- `XXXRoomHandle`
- `XXXRoomModel`

## 18. 脚本粒度与反碎片化规范

### 18.1 脚本粒度必须服务于业务边界理解

脚本拆分的目标，是让开发者更快看清业务边界，而不是让文件数量看起来很多。

### 18.2 共享契约层协议禁止一协议一文件式过度拆分

共享契约层中的业务协议，不允许默认采用“一条协议一个脚本”的拆分方式。

### 18.3 推荐协议脚本聚合粒度

推荐采用以下聚合规则：

- 一个全局模块的所有协议，放在一个协议聚合脚本中
- 一个房间业务组件的所有协议，放在一个协议聚合脚本中

### 18.4 Handle / Model 仍保持独立主脚本

Handle 与 Model 属于业务主入口和主状态承载层，应保持为独立主脚本。

### 18.5 局部类型允许适度聚合

局部事件、局部快照、局部配置、局部辅助结构等类型，允许根据业务边界适度聚合到少量脚本中。

## 19. 服务端基础设施规范

### 19.1 GlobalInfrastructure 的定位

`GlobalInfrastructure` 是服务端顶级装配门面。它负责：

- 全局配置装载
- 统一时间源驱动
- `MessageRegistry` 构建
- 网络适配器装配
- 会话管理器装配
- 房间管理器装配
- 全局业务单元装配
- 全局业务单元生命周期驱动
- Tick 驱动与关停清理

### 19.2 Infrastructure 只装配，不展开业务监听细节

`GlobalInfrastructure` 只负责：

- 创建 Handle
- 创建 Model
- 为 Handle 注入依赖
- 调用 Handle 统一注册入口
- 驱动启动、运行、关停

它不应直接展开各 Handle 监听了哪些协议，也不应堆积匿名注册逻辑。

### 19.3 GlobalEventBus 的物理归属

`GlobalEventBus` 必须由 `GlobalInfrastructure` 持有，并与 `GlobalScope ServiceLocator` 处于同一服务端全局作用域层级。

强制约束如下：

1. `GlobalEventBus` 属于服务端全局基础设施域，不属于 `GlobalRoomManager`
2. `GlobalEventBus` 的生命周期由 `GlobalInfrastructure` 统一装配、统一清理
3. `GlobalEventBus` 只负责服务端全局域模块之间的领域事件传播
4. `GlobalEventBus` 不承担房间域跨房间事件总线职责
5. 房间域结果若需上升到服务端全局域，必须由明确的 Handle 或 Service 转译后再进入 `GlobalEventBus`
6. `GlobalEventBus` 不承担网络协议广播职责

### 19.4 GlobalRoomManager 的全局服务归属

`GlobalRoomManager` 必须作为服务端核心全局服务之一，注册到 `GlobalInfrastructure` 的 `GlobalScope ServiceLocator` 中。

### 19.5 RoomDispatcher 与 GlobalRoomManager 的职责切割

#### RoomDispatcher

只作为**全局业务入口协调服务**，负责：

1. 处理建房、加房、离房等全局请求
2. 执行参数校验、权限校验、幂等判断与流程协调
3. 维护用户 / 会话 / 房间的全局关系语义
4. 在需要时调用 `GlobalRoomManager` 执行房间创建、查找、销毁或成员调度动作
5. 在建房流程中构建并准备该房间的具体 `IRoomSettings`

#### GlobalRoomManager

只作为**房间运行时物理宿主与调度核心**，负责：

1. 持有 `RoomInstance` 集合
2. 创建、注册、移除 `RoomInstance`
3. 驱动所有房间 Tick
4. 维护可路由房间集合
5. 执行房间空置巡检与生命周期收尾
6. 作为全局模块访问房间运行时能力的统一代理门面

强制约束如下：

- `RoomDispatcher` 不直接作为 `RoomInstance` 集合的物理持有者
- `GlobalRoomManager` 不承担建房 / 加房业务入口 Handle 职责
- 大厅请求协调与房间物理宿主职责不得揉合为同一个对象
- 全局模块不得直接穿透目标房间的 `RoomScope ServiceLocator` 获取内部服务
- 全局模块若需要查询房间运行时信息，必须通过 `GlobalRoomManager` 提供的公开代理查询接口完成

### 19.6 服务端宿主驱动约束

在 Unity 环境中，服务端顶级宿主应遵循如下驱动方式：

1. 宿主层调用 `GlobalInfrastructure.Initialize()`
2. 每帧调用 `GlobalInfrastructure.Tick()` 或等价主循环入口
3. 关停时调用 `GlobalInfrastructure.Shutdown()`
4. 宿主层不得绕过 `GlobalInfrastructure` 分别直接驱动多个全局服务主循环
5. 房间级主循环由 `GlobalRoomManager` 承担，但其驱动入口仍必须经由 `GlobalInfrastructure` 统一协调

### 19.7 Shutdown 关停顺序约束

`GlobalInfrastructure.Shutdown()` 必须按以下顺序执行：

1. 停止接收新连接
2. 向所有在线客户端广播服务器关闭通知
3. 停止接收新消息，并允许当前帧已进入处理链的消息完成
4. 阻止新房间创建与新业务请求进入
5. 触发所有 `RoomInstance` 的销毁流程
6. 等待录像文件收尾与落盘完成，可设置明确超时上限
7. 按初始化逆序销毁全局服务
8. 最后关闭 Adapter

## 20. 客户端基础设施规范

### 20.1 ClientInfrastructure 的定位

`ClientInfrastructure` 是客户端顶级装配门面。它负责：

- 构建 `MessageRegistry`
- 构建 `Serializer`
- 构建 `ClientNetworkAdapter`
- 构建 `SessionContext`
- 构建 GlobalRouter / RoomRouter
- 构建 NetworkEntry / MessageSender / ReplayController
- 装配客户端业务单元
- 驱动在线模式与回放模式切换
- 驱动生命周期关停

### 20.2 ClientInfrastructure 不得承担业务消息细节注册职责

`ClientInfrastructure` 只负责装配与驱动，不负责展开每个 Handle 的协议监听清单。

### 20.3 客户端全局运行时门面定位

客户端必须存在一个全局运行时协调门面，例如 `GlobalClientManager`。其负责范围包括：

1. 持有客户端主状态机
2. 持有底层连接状态字段
3. 持有当前在线 `ClientRoomInstance`
4. 直接字段持有 `ReplayPlaybackController`
5. 协调进入房间、退出房间、异常断开与回放切换
6. 作为客户端全局基础设施模块的统一装配与寻址入口
7. 协调 `ClientNetworkEntry`、客户端状态协议过滤器与 `ReplayPlaybackController`
8. 直接持有少量必要的客户端全局子模块或客户端发送器
9. 作为客户端房间创建、恢复与销毁流程的唯一协调入口

其明确不负责以下内容：

1. 服务端式多房间集合调度
2. 房间权威 Tick 管理
3. 运行期多房间可路由集合维护
4. 客户端具体 UI、View 或项目表现细节实现

### 20.4 客户端全局子模块持有策略

客户端不强制建立与服务端完全对称的全局 `ServiceLocator` 体系。默认制度如下：

1. 客户端全局层以 `GlobalClientManager` 为统一门面
2. 少量必要的客户端全局子模块由 `GlobalClientManager` 直接字段持有或显式装配持有
3. 除非项目复杂度显著上升，否则不建议在客户端再复制一套服务端式全局服务容器结构
4. 若项目后续确实扩展出复杂客户端全局模块体系，也不得破坏当前 `GlobalClientManager` 作为统一生命周期门面的地位

## 21. 客户端状态机与协议过滤规范

### 21.1 客户端主状态机转移表

| 当前状态 | 触发事件/源 | 目标状态 | 说明 |
|---|---|---|---|
| `Disconnected` | 成功建立底层网络连接 | `Authenticating` | 开始登录握手。本地是否持有有效 `SessionId` 由开发者在此阶段判断，决定发送 `C2S_Login` 还是 `C2S_Reconnect` |
| `Authenticating` | 收到 `S2C_LoginResult(Success)` | `InLobby` | 首次登录成功，服务端下发新 `SessionId` |
| `Authenticating` | 收到 `S2C_ReconnectResult(Success)` | `InLobby` 或 `InRoom` | 重连成功。若服务端判定原会话在房间内，则通过全局客户端门面的显式恢复装配协调流程进入房间恢复路径 |
| `Authenticating` | 收到 `S2C_ReconnectResult(Fail)` | `Disconnected` | 重连凭证无效，退回登录入口 |
| `Authenticating` | 底层连接断开 | `Disconnected` | 框架不自动发起重连 |
| `InLobby` | 收到 `S2C_JoinRoomResult(Success)` | `InRoom` | 成功加入房间，进入正常入房装配流程 |
| `InRoom` | 收到 `S2C_RoomDismissed` 或主动离开 | `InLobby` | 房间销毁或退出，清理局内实例 |
| `InLobby` / `InRoom` | 触发本地回放加载 | `InReplay` | 进入回放前必须先销毁当前在线 `ClientRoomInstance`，再创建本地回放房间 |
| `InReplay` | 回放结束或主动退出 | `InLobby` | 默认安全策略是退回大厅态 |
| `Disconnected` / `InLobby` / `InRoom` | 底层连接断开 | `Disconnected` | 在线运行态的强制兜底 |
| `InReplay` | 底层连接断开 | `InReplay` | 只更新在线连接状态，不强制中断本地回放房间 |

### 21.2 主状态机与底层连接状态分离

客户端除维护主状态机外，还必须维护一个独立的**底层连接状态字段**，用于表示当前网络连接健康度。

建议至少包含以下取值：

- `Connected`
- `Disconnected`

约束如下：

1. 主状态机用于表达当前应用所处阶段，例如 `InLobby`、`InRoom`、`InReplay`
2. 底层连接状态字段用于表达当前在线连接是否有效
3. 二者并行维护，互不等价
4. 在 `InReplay` 状态下，底层连接状态变化不得直接摧毁本地回放房间生命周期

### 21.3 非法协议与非法迁移阻断表

客户端状态机除正常迁移外，必须显式定义非法消息边界：

1. 每个状态明确允许接收哪些 `S2C_XXX`
2. 每个状态明确允许发送哪些 `C2S_XXX`
3. 收到不属于当前状态可处理集合的消息时，必须执行“报错并丢弃”或“报错后回退到安全状态”的固定策略

强制约束如下：

- `Disconnected` 状态下不得处理任何房间域广播
- `Authenticating` 状态下不得处理任何局内业务事件
- `InRoom` 状态下再次收到登录成功结果时，必须视为非法协议并阻断
- 房间销毁广播与底层断开同时出现时，以底层断开后的 `Disconnected` 为最终兜底状态

### 21.4 客户端状态协议过滤器

客户端必须提供统一的**客户端状态协议过滤器**。

强制约束如下：

1. 在协议正式进入任意 Router 前，根据当前客户端状态检查该协议是否允许接收
2. 在客户端准备发包前，根据当前客户端状态检查该协议是否允许发送
3. 非法协议统一执行“报错并阻断”或“报错并回退安全状态”的策略
4. 所有状态允许集与禁止集必须集中注册，禁止分散在各业务 Handler 内部维护
5. 客户端状态协议过滤器必须位于 `ClientNetworkEntry` 与各类 Router 之间
6. 未通过状态过滤的协议不得进入 `ClientGlobalMessageRouter` 或 `ClientRoomMessageRouter`
7. 严禁将状态过滤责任下沉到各个具体 Handler 内部各自判断

### 21.5 断线后的延迟消息污染拦截

- 当前不存在有效 `ClientRoomInstance` 时，所有房间域 `S2C_XXX` 默认直接丢弃
- 当前状态不是 `InRoom` 时，所有在线房间域广播默认直接丢弃
- 当前状态是 `InReplay` 时，所有在线房间域广播必须直接丢弃
- 延迟消息不得重新驱动已销毁 View、已释放组件或已退出房间的本地状态

### 21.6 InReplay 边界隔离与事件分流

1. 进入 `InReplay` 状态时，客户端严格禁止创建或加入任何在线房间
2. 在 `InReplay` 状态下，客户端直接丢弃所有在线房间域 `S2C_XXX`
3. 允许继续接收部分全局域消息，但仅限进入**静态协议类型白名单**的消息
4. 白名单职责归属：必须由客户端状态协议过滤器集中注册维护
5. 未进入白名单的全局域消息，在 `InReplay` 状态下默认阻断
6. 底层连接是否断开由开发者根据业务决策，框架提供模式切换钩子

### 21.7 退出回放的默认迁移策略

退出回放模式后的框架默认安全策略为**回到 `InLobby`**。

约束如下：

1. 框架默认不自动恢复进入回放前的在线房间态
2. 即使底层在线连接在回放期间保持存活，退出回放后默认仍回到 `InLobby`
3. 若项目需要恢复到进入回放前的在线态，属于开发者扩展策略，框架不默认承担该恢复编排

### 21.8 大厅态是进入回放的唯一合法入口态

客户端进入回放模式前，必须处于大厅态。大厅态表示当前不存在激活中的在线房间上下文，也不存在正在执行中的创建房间、进入房间或房间切换流程。

## 22. 全局业务单元统一生命周期规范

### 22.1 全局业务单元定义

全局业务单元是挂在全局域、依赖 `GlobalInfrastructure` 存活的长期运行结构，例如：

- `UserHandle + UserModel`
- `RoomDispatcherHandle + RoomDispatcherModel`
- `ReconnectHandle + ReconnectModel`
- `AnnouncementHandle + AnnouncementModel`
- `LobbyGlobalChatHandle + LobbyGlobalChatModel`
- `ReplayHandle + ReplayModel`

### 22.2 生命周期阶段

全局业务单元必须统一具备以下阶段语义：

1. 创建 Model
2. 创建 Handle
3. 依赖注入
4. 统一注册消息监听
5. 运行
6. 注销监听
7. 释放

### 22.3 生命周期与监听注册绑定

全局业务单元的消息监听必须与其生命周期绑定。禁止业务单元停用后仍保留监听。

## 23. 房间实例与房间业务单元生命周期统一规范

### 23.1 RoomInstance 是房间域唯一生命周期源

`RoomInstance` 是单个房间的唯一生命周期总入口。房间业务组件不拥有独立主生命周期，它们的生命周期完全来自 `RoomInstance` 的统一驱动。

### 23.2 房间实例运行态与房间业务态必须区分

`RoomInstance` 运行态用于表达框架对象的存活阶段，例如：

- `Initializing`
- `Running`
- `Destroying`
- `Destroyed`

这套运行态只用于框架对象管理，不等于房间业务主生命周期。

### 23.3 房间业务主生命周期

房间业务主生命周期统一定义为：

1. 房间创建
2. 等待开始
3. 游戏中
4. 游戏结束处理中
5. 结算中
6. 房间销毁

制度要求如下：

- 录制边界必须挂靠房间业务主生命周期，而不是仅看 `RoomInstance` 是否处于 `Running`
- 重连恢复边界必须挂靠房间业务主生命周期
- 业务单元的阶段行为必须挂靠房间业务主生命周期
- 回放文件中的业务阶段语义，也应以该主生命周期为准

### 23.4 Tick 与业务态的关系

房间实例从创建成功后即可进入持续 Tick 驱动，并一直运行到房间销毁。但 Tick 持续存在不代表房间始终处于“正式对局进行中”。

### 23.5 房间业务单元结构规范

房间业务单元统一组织为：

- `XXXRoomHandle`
- `XXXRoomModel`

其中：

- RoomHandle 负责房间生命周期回调接入、注册与处理消息、组织业务逻辑、处理恢复流程
- RoomModel 负责保存房间业务状态与可恢复数据

### 23.6 房间业务单元没有独立主生命周期

房间业务单元自身不定义独立的业务生命周期阶段。它只响应 `RoomInstance` 的统一阶段回调。

### 23.7 房间实例统一驱动业务组件的阶段回调

`RoomInstance` 应统一向房间业务单元分发阶段回调，至少包括以下语义：

- `OnRoomCreate`
- `OnRoomWaitStart`
- `OnRoomStartGame`
- `OnRoomGameEnding`
- `OnRoomSettling`
- `OnTick`
- `OnRoomDestroy`

### 23.8 生命周期调用关系示意

```text
OnRoomCreate():
    _components.OnRoomCreate()

OnRoomWaitStart():
    _components.OnRoomWaitStart()

OnRoomStartGame():
    _components.OnRoomStartGame()

OnRoomGameEnding():
    _components.OnRoomGameEnding()

OnRoomSettling():
    _components.OnRoomSettling()

OnTick():
    _components.OnTick()

OnRoomDestroy():
    _components.OnRoomDestroy()
```

### 23.9 房间基础设置组件必须在创建房间时明文挂载

创建房间流程中，必须明文写死挂载房间基础设置组件。该挂载行为属于房间实例创建成功所必需的骨架装配步骤，不允许被配置关闭，不允许遗漏。

### 23.10 基础空房间的定义边界

“基础空房间”是指**不内置任何业务玩法组件**的最小房间运行时容器，而不是完全不含框架基础设施的空对象。

基础空房间至少具备：

- `RoomId`
- 房间生命周期状态
- 成员身份集合
- 当前在线连接映射
- `RoomScope ServiceLocator`
- `RoomEventBus`
- 房间内 Tick 计数器
- `ReplayRecorder` 挂载位
- `IRoomSettings`
- `ServerRoomMessageRouter`

### 23.11 成员身份集合与在线连接映射分离

`RoomInstance` 必须显式区分以下两类运行时集合：

1. **成员身份集合**
   - 表示哪些会话 / 成员逻辑上属于该房间
   - 断线后仍可保留其成员归属

2. **当前在线连接映射**
   - 表示当前有哪些有效连接在线附着到这些成员
   - 重连接管时可发生替换

强制约束如下：

1. 断线后成员仍可属于房间，但可不在当前在线连接映射中
2. 房间空置超时判断针对“当前在线连接映射是否为空”
3. 房间成员归属判断针对“成员身份集合”
4. 广播目标计算默认基于当前在线连接映射，不基于纯成员身份集合
5. 不得以一个模糊的单一字典同时混合承担成员归属与在线连接状态两种语义

### 23.12 重复销毁保护

`RoomInstance` 最少包含：

- `Active`
- `Destroying`
- `Destroyed`

一旦进入 `Destroying`，后续重复销毁请求只允许输出 Warning 并立即返回。

### 23.13 Destroying 状态下的路由移除规则

`RoomInstance` 一旦进入 `Destroying` 状态，必须立即从 `GlobalRoomManager` 的**可路由房间集合**中移除。

### 23.14 Destroying 状态允许行为边界

房间一旦进入 `Destroying` 状态，运行时必须严格收缩为**销毁链与收尾链**。

强制约束如下：

1. 禁止再接收任何新的协议输入
2. 禁止新的业务状态修改
3. 禁止新的业务广播发送
4. 仅允许组件反初始化、总线清理、录像收尾、资源释放与必要的销毁日志输出继续执行
5. 若某组件在 `Destroying` 期间继续尝试发起新的业务流程，必须直接报错并阻断

## 24. 房间装配器与 Router 动态注册规范

### 24.1 服务端房间装配器制度

服务端房间业务组件装配必须通过**专职房间装配器**完成，例如 `ServerRoomAssembler` 或等价装配器。

强制约束如下：

1. 房间装配逻辑不得直接内嵌在 `RoomInstance` 本体中
2. `GlobalRoomManager` 或 `RoomDispatcher` 可触发房间装配流程，但不直接承担装配细节实现
3. `ServerRoomAssembler` 负责组件实例化、依赖接线、Handler 能力收集、Router 绑定与装配失败阻断
4. `RoomInstance` 作为运行时容器，不承担装配流程编排职责
5. 任一房间业务组件实例化失败、初始化失败或 Router 绑定失败时，必须中止整个房间装配流程并执行原子回滚

### 24.2 ServerRoomMessageRouter 生命周期与注册时机

`ServerRoomMessageRouter` 必须在 `RoomInstance` 创建时随房间实例一起创建，但其内部 Handle 注册表不得预注册所有可能的房间业务 Handle。

强制约束如下：

1. `ServerRoomMessageRouter` 作为房间内基础路由容器，在房间创建时即存在
2. 具体房间业务 Handle 必须随房间业务组件装配过程动态注册
3. 房间业务组件销毁或卸载时，必须同步动态注销其所持有的协议处理绑定
4. 不允许在房间创建阶段预注册“所有理论上可能存在的房间业务 Handle”，再通过激活开关做伪动态管理
5. 同一 `RoomInstance` 内，同一协议类型只能存在一个主处理 Handle
6. 若动态注册时发现同一协议类型已存在其他主处理者，必须直接报错并阻断当前房间装配流程
7. 动态注销时必须校验注销来源与原注册来源一致，防止误删其他组件绑定
8. Router 实例可稳定存在，但其可处理协议集合必须与当前实际已装配的房间业务组件集合严格一致
9. `ServerRoomMessageRouter` 的注册 / 注销发起方必须是 `ServerRoomAssembler` 或等价专职房间装配器
10. 房间业务组件自身不得在 `Init()`、`OnDestroy()` 或等价生命周期中主动直接调用 Router 完成注册 / 注销
11. 房间业务组件只负责声明其可处理的协议能力，统一装配流程负责将该能力绑定到 Router

### 24.3 服务端房间装配失败回滚约束

`ServerRoomAssembler` 在装配中途失败时，必须执行**原子性回滚**，不得留下半装配状态。

强制约束如下：

1. 已成功绑定到 `ServerRoomMessageRouter` 的 Handle，必须按装配逆序注销
2. 已完成初始化的房间业务组件，必须按装配逆序执行反初始化
3. 已实例化但未完成初始化的房间业务组件，必须按装配逆序销毁或释放
4. 任一回滚步骤失败时，必须输出详细 Error 日志，但仍继续执行后续回滚，确保最终不留脏状态
5. 只有在回滚完成后，才允许向上层返回房间装配失败结果

### 24.4 客户端房间装配器归属约束

客户端房间业务组件装配必须通过**实例级房间装配器**完成，例如 `ClientRoomAssembler` 或等价装配器。

强制约束如下：

1. 客户端房间装配器的生命周期必须绑定到具体 `ClientRoomInstance`
2. 客户端全局门面只负责协调触发客户端房间装配流程，并在预装配阶段创建空壳 `ClientRoomInstance` 与 `ClientRoomAssembler`
3. `ClientRoomAssembler` 在持有目标 `ClientRoomInstance` 上下文的前提下执行组件装配、初始化与 Router 绑定
4. 客户端房间装配器不得作为全局常驻单例跨多个房间实例复用上下文
5. 在线房间装配与本地回放房间装配都必须使用各自实例上下文下的装配器执行
6. 任一客户端房间业务组件实例化失败、初始化失败或 Router 绑定失败时，必须中止整个客户端房间装配流程
7. 只要任一组件装配失败，客户端状态不得切入 `InRoom` 或进入可运行的回放房间态

### 24.5 ClientRoomMessageRouter 生命周期与注册时机

`ClientRoomMessageRouter` 必须在 `ClientRoomInstance` 创建时随房间实例一起创建，但其内部 Handle 注册表不得预注册所有可能的客户端房间业务 Handle。

强制约束如下：

1. `ClientRoomMessageRouter` 作为客户端房间内基础路由容器，在房间实例创建时即存在
2. 具体客户端房间业务 Handle 必须随房间业务组件装配过程动态注册
3. 房间业务组件销毁、卸载或退出回放时，必须同步动态注销其所持有的协议处理绑定
4. 不允许在客户端房间创建阶段预注册“所有理论上可能存在的客户端房间业务 Handle”，再通过激活开关做伪动态管理
5. 同一 `ClientRoomInstance` 内，同一协议类型只能存在一个主处理 Handle
6. 若动态注册时发现同一协议类型已存在其他主处理者，必须直接报错并阻断当前房间装配流程
7. 动态注销时必须校验注销来源与原注册来源一致，防止误删其他组件绑定
8. Router 实例可稳定存在，但其可处理协议集合必须与当前实际已装配的客户端房间业务组件集合严格一致
9. `ClientRoomMessageRouter` 的注册 / 注销发起方必须是 `ClientRoomAssembler` 或等价专职客户端房间装配器
10. 客户端房间业务组件自身不得在 `Init()`、`OnDestroy()` 或等价生命周期中主动直接调用 Router 完成注册 / 注销
11. 客户端房间业务组件只负责声明其可处理的协议能力，统一装配流程负责将该能力绑定到 Router

## 25. 房间路由隔离与 RoomId 规范

### 25.1 每个房间实例必须拥有独立 RoomId 路由

每个 `RoomInstance` 都必须独立拥有属于自己的 `RoomId` 路由上下文。10 个房间就应当存在 10 套彼此隔离的房间路由上下文。

### 25.2 绝不允许串房消息

不允许出现以下情况：

- 第一个房间收到第二个房间的消息
- 第二个房间收到第三个房间的广播
- 客户端携带错误 `RoomId` 后仍能进入其他房间业务单元处理链
- 同一个连接凭错误 `RoomId` 穿透到非本房间实例

### 25.3 RoomId 路由校验链必须严格执行

房间消息必须同时通过以下校验：

- `ConnectionId` 有效
- 会话存在
- 会话当前处于房间中
- `Envelope.RoomId` 与 `Session.CurrentRoomId` 完全一致
- `GlobalRoomManager` 只能路由到对应 `RoomId` 的房间实例

任一失败都必须立即丢弃消息。

### 25.4 服务端接收链的域路由规则

服务端收到 `NetworkEnvelope` 后，必须按以下顺序处理：

1. 根据 `MessageId` 查询 `MessageRegistry`
2. 获取对应协议类型并完成反序列化
3. 根据四协议基类判断方向与域归属：
   - `C2SGlobalMessage` → 进入全局模块入口
   - `C2SRoomMessage` → 进入房间域路由入口
4. 只有在确认协议属于 `C2SRoomMessage` 之后，才允许进入房间归属一致性校验链
5. 对 `C2SGlobalMessage` 禁止误入房间归属一致性校验链
6. 若服务端入站链收到被注册为 `S2CGlobalMessage` 或 `S2CRoomMessage` 的协议类型，必须视为非法协议并立即阻断

### 25.5 服务端房间域消息的强制校验链

所有房间域 `C2S_XXX` 消息在进入 `RoomInstance` 业务组件前，必须固定执行以下前置校验：

1. 当前消息已被判定为 `C2SRoomMessage`
2. `ConnectionId` 是否有效
3. `ConnectionId` 是否已绑定有效会话
4. 会话是否存在 `CurrentRoomId`
5. 运行时房间上下文中的 `RoomId` 是否与 `CurrentRoomId` 完全一致

任意一步失败时，框架必须：

- 直接输出详细 Error 日志
- 立即丢弃当前消息
- 严禁继续向房间业务组件派发

### 25.6 RoomId 不是随意字符串

`RoomId` 是房间隔离、消息路由、重连恢复、录制归档的关键上下文标识。

### 25.7 RoomId 统一生成规则

`RoomId` 明文规定采用以下组合生成：

- 当前实时时间戳
- 随机码
- 房间名

### 25.8 禁止的 RoomId 生成方式

不允许采用以下低信息量生成方式作为正式规范：

- 单纯自增号
- 单纯时间戳
- 单纯房间名
- 单纯随机串
- 不可读且不可诊断的黑盒 GUID 拼接后无业务标识

## 26. 会话与连接语义规范

### 26.1 ConnectionId 与 SessionId 边界

- `ConnectionId` 表示当前底层连接标识
- `SessionId` 表示服务端签发的会话标识

断线后 `ConnectionId` 可以失效，`SessionId` 可以保留用于重连恢复。

### 26.2 Session 重连接管规则

框架必须制度化同一 `SessionId` 的连接接管行为：

1. 任意时刻，一个 `SessionId` 只允许存在一个有效主连接
2. 新连接完成认证或重连认证成功后，旧连接必须立即标记为 `Replaced`
3. 被标记为 `Replaced` 的旧连接，其后续所有来包一律拒收
4. 若会话当前位于房间内，必须先完成房间连接映射替换，再抛出重连事件
5. 重连事件只表达“连接接管已完成”，不承载任何业务快照恢复逻辑

### 26.3 ClientSessionContext 状态语义

必须继续保留并强调：

- `HasRoomBinding` 只表示曾经绑定过房间
- `IsInRoom` 才表示当前在线且可正常进行房间通信

禁止业务仅凭 `CurrentRoomId` 非空就判定当前可进行房间通信。

## 27. 断线重连机制规范

### 27.1 重连机制定位

重连是原会话与原房间上下文的权威恢复流程，不是重新登录一次。

### 27.2 标准流程

统一重连流程应为：

1. 客户端发起重连请求
2. 服务端校验 `SessionId` 并完成会话接管
3. 服务端恢复该会话的房间在线映射
4. 服务端向客户端下发房间业务单元清单
5. 客户端根据业务单元清单重建本地房间结构
6. 服务端通知房间实例进入恢复协调流程
7. 房间实例通知各房间业务单元准备恢复数据
8. 各业务单元向该重连客户端提供自身最新快照
9. 客户端完成状态恢复后回到正常流程

### 27.3 接入要求

所有房间业务单元必须支持统一重连恢复。这不是可选项，而是接入门槛。

### 27.4 Handle-Model 恢复职责边界

- RoomHandle 负责恢复流程组织与恢复消息处理
- RoomModel 负责承载业务单元自身需要恢复的状态数据

### 27.5 ReconnectResult 的协议归属与载荷边界

`S2C_ReconnectResult` 始终属于**全局域协议**，即使其在恢复到房间态时携带房间相关信息，也不因此转化为房间域协议。

当目标状态为房间内时，`S2C_ReconnectResult` 允许直接内嵌：

- `RoomId`
- 房间业务组件清单
- 进入房间装配流程所需的最小框架级字段

约束如下：

1. 这些字段只用于帮助客户端确定目标状态并执行房间装配预处理
2. 这些字段不等价于业务组件状态快照
3. 具体业务状态恢复仍由开发者后续自行实现
4. 框架不强制要求再拆分额外的“重连房间恢复协议”

### 27.6 ReconnectResult 的客户端恢复协调边界

`S2C_ReconnectResult` 即使携带房间恢复相关字段，仍然属于全局域协议，只能进入 `ClientGlobalMessageRouter`。

强制约束如下：

1. 处理 `S2C_ReconnectResult` 的客户端重连模块 Handle，不得直接创建、销毁或操作 `ClientRoomInstance`
2. 若 `S2C_ReconnectResult` 的目标状态为房间内，必须通过客户端全局门面的显式恢复装配协调接口触发后续房间恢复流程
3. `S2C_ReconnectResult` 中携带的 `RoomId` 属于业务字段，用于表达目标恢复房间
4. 该业务字段与 `NetworkEnvelope.RoomId` 的运行时上下文语义彼此独立，不得混淆

## 28. 房间录制业务单元与 Replay 规范

### 28.1 录制能力的定位

房间录制能力必须被定义为房间域中的标准业务单元能力，不能被视为发送链顺手附带的一段零散逻辑。

### 28.2 ReplayRecorder 的挂载策略

`ReplayRecorder` 属于房间可选挂载的回放录制能力，而不是所有房间的强制固定组件。

约束如下：

1. 基础空房间必须具备 `ReplayRecorder` 挂载位
2. 只有当前房间策略启用录制时，才在房间创建阶段申请专属录像通道并挂载有效 `ReplayRecorder`
3. 未启用录制时，`ReplayRecorder` 挂载位允许为空
4. 服务端发送协调节点在发现当前房间未挂载有效 `ReplayRecorder` 时，必须直接跳过录制分支，不得影响正常发送流程

### 28.3 录制只在游戏中阶段生效

服务端正式录制只允许在房间业务生命周期的“游戏中”阶段生效。等待开始阶段不纳入正式录制。

### 28.4 当前公共广播不等于合法录制

即便某些等待开始阶段发生了公共广播，也不得自动视为正式可录制事件。录制与否必须由房间业务生命周期裁定。

### 28.5 回放录制源的范围约束

录像系统只允许录制以下消息：

- 房间域
- `S2C`
- 发送给全体房间成员的公共广播消息

录像系统明确不录制以下内容：

1. 全局域消息
2. 房间域的单体成员单播消息
3. 客户端本地临时事件
4. 客户端本地预测结果
5. 纯 UI 层本地表现逻辑

### 28.6 回放公共历史原则

回放系统重建的是某一房间在录制期间的**公共状态演进过程**。因此回放依据必须严格来源于服务端曾经发往**全体房间成员**的权威 `S2C` 结果流。

### 28.7 ReplayRecorder 的自动旁路拦截点约束

`ReplayRecorder` 的写入触发点必须固定在**服务端房间公共广播发送管线出口**，由框架自动旁路拦截完成，开发者无需也不得手动补录。

仅当一条消息同时满足以下条件时，才允许进入录制：

1. 消息属于房间域
2. 消息方向为 `S2C`
3. 投递范围为“全体房间成员公共广播”
4. 当前房间业务主生命周期处于“游戏中”

### 28.8 ReplayRecorder 的写入模型约束

`ReplayRecorder` 的“旁路写入”必须采用**主线程生产、异步队列消费**的模型。

约束如下：

1. 主线程只负责将待录制数据写入内存缓冲队列
2. 后续落盘由独立写入流程异步消费
3. 任意时刻不得因录像写入阻塞主线程
4. 当缓冲达到上限时，按既定背压策略执行“丢弃最旧帧并输出 Warning”

### 28.9 缓冲背压机制

- 缓冲背压依赖 `ReplayBufferCapacity`
- 溢出策略采用**丢弃最旧帧 + 输出 Warning**
- 绝不阻塞主线程

## 29. 回放文件结构规范

### 29.1 .sfr 文件头必须包含

`.sfr` 回放文件头必须明确包含：

- `FrameworkVersion`
- `ProtocolVersion`
- `SettingsFormat`
- `SettingsVersion`
- `SettingsPayload`
- `RoomComponentIds`
- `TotalTicks`
- `RecordStartUnixMs`
- `ContentMd5`

### 29.2 回放记录结构约束

`.sfr` 文件记录的是**面向回放的标准化回放记录结构**，不是线上传输封套原样落盘。

每条回放记录至少应包含：

- `Tick`
- `MessageId`
- `Payload`
- `RoomId`

约束如下：

1. 回放记录结构不得直接依赖线上传输封套的完整字节布局
2. 在线传输头部调整不得被默认视为自动兼容历史回放文件
3. 回放落盘结构必须服务于重演稳定性，而不是图省事复用在线原始包

### 29.3 RoomComponentIds 文件头约束

`.sfr` 文件头必须显式写入 `RoomComponentIds`，并作为回放房间装配依据。

强制约束如下：

1. `RoomComponentIds` 中保存的是稳定组件注册标识，不得使用运行时类型名
2. 回放房间装配所需的房间业务组件清单不得依赖扫描录制帧后反推
3. 即使某房间业务组件在录制期间没有产生任何公共广播帧，只要其属于录制时房间结构的一部分，也必须出现在 `RoomComponentIds` 中
4. `RoomComponentIds` 的数组顺序即回放房间的组件装配顺序
5. 回放加载时若发现 `RoomComponentIds` 中存在本地未注册组件，必须直接报错并阻断回放装配
6. 不允许因“录制期间未发帧”而遗漏本应存在的房间业务组件

### 29.4 MD5 完整性校验

1. 仅对除文件头之外的录像数据块进行哈希计算
2. 客户端在完整接收所有分块并本地拼接完成后，立即计算 MD5 并与文件头 `ContentMd5` 比对
3. 校验失败则直接丢弃文件并输出 Error

### 29.5 分块传输字段约定

Replay 分块协议必须至少携带：

- `ReplayId`
- `ChunkIndex`
- `TotalChunks`
- `PayloadLength`
- `TotalLength`

框架只负责：

1. 缓存分块
2. 判断是否收齐
3. 收齐后按顺序拼装
4. 拼装完成后执行一次终态 MD5 校验

## 30. 客户端回放重演控制规范

### 30.1 回放控制器定位

`ClientReplayPlaybackController` 是客户端回放域中的本地重演控制模块。它负责加载回放文件、驱动时间线、按录制顺序投喂回放事件，并以本地模拟服务端下行事件的方式驱动客户端房间表现层完成重演。

### 30.2 ReplayPlaybackController 的物理归属

`ReplayPlaybackController` 属于客户端**全局级回放基础设施控制器**，由客户端全局门面直接字段持有。

约束如下：

1. `ReplayPlaybackController` 不属于 `ClientRoomInstance` 内部组件
2. 其生命周期高于单个在线房间实例
3. 其负责在线态与回放态切换协同
4. 其负责创建、驱动与销毁本地回放房间
5. 其不得依赖某个在线 `ClientRoomInstance` 存活后才成立

### 30.3 本地回放房间定位

- 本地回放房间仅存在于客户端内存中
- 不参与任何在线网络通信
- 只消费本地回放数据流
- 与在线 `ClientRoomInstance` 必须逻辑隔离、实例隔离、事件隔离

### 30.4 进入回放模式的创建流程

1. 前置检查回放文件是否存在、`FrameworkVersion` 是否匹配、`ProtocolVersion` 是否匹配、`ContentMd5` 是否有效
2. 若当前存在在线 `ClientRoomInstance`，必须先执行完整销毁流程
3. 创建本地回放房间实例
4. 根据回放文件头中的房间配置元数据与 `SettingsPayload` 还原本地回放房间基础配置
5. 根据 `RoomComponentIds` 按稳定顺序装配本地回放房间所需组件
6. 建立回放时间轴、Tick 游标与播放状态机
7. 开始按回放数据直接驱动本地回放房间的 `ClientRoomMessageRouter` 重演

### 30.5 回放重演主轴约束

Replay 重演必须以 **Tick** 为主轴。

约束如下：

1. 回放记录必须能够定位所属 Tick
2. 虚拟时钟只负责控制 Tick 推进速度、播放速率与跳转调度
3. 录制文件中的 Tick 明确表示**房间内相对 Tick**，不是服务端全局 Tick
4. `RecordStartUnixMs` 主要用于文件记录、诊断与展示，不作为主重放索引
5. 快进、慢放、暂停、恢复与跳转的底层执行，最终都应落实为 Tick 游标控制
6. 框架只保证“从起点按 Tick 顺序重演到目标位置”的正确性，不强制提供检查点快照跳转优化

### 30.6 回放时间轴驱动源

`ReplayPlaybackController` 必须使用独立的**回放虚拟时钟**驱动。

支持：

- 播放
- 暂停
- 恢复
- 快进
- 慢放
- 跳转
- 回放结束通知
- 主动退出回放

### 30.7 回放运行期隔离规则

1. 在线房间事件不得进入本地回放房间
2. 本地回放房间事件不得反向写入在线房间
3. 回放状态下不得创建或加入在线房间
4. 回放中的组件实例不得复用在线组件实例引用
5. 回放控制器不得依赖在线房间存活

### 30.8 回放与进入房间行为互斥

进入回放与进入房间、创建房间属于互斥行为。二者分别建立不同的当前房间域上下文，不允许同时作为激活态存在。

### 30.9 回放不参与在线权威

回放控制器不参与服务端权威状态裁定，不反向影响在线房间状态。

### 30.10 回放事件来源定义

回放模式下，客户端房间域的数据来源不是服务端实时网络下行，而是本地回放控制器根据回放文件内容，按录制顺序与时间线模拟服务端事件进行重演。

### 30.11 退出回放模式的完整清理机制

1. 停止回放时间轴推进
2. 切断本地回放房间后续消息注入
3. 通知所有回放组件执行反初始化逻辑
4. 触发本地回放房间的 `EventBus` 清理、表现层上下文挂载位清理钩子与房间作用域服务清理
5. 释放回放文件缓存、分块缓存、时间轴缓存、索引缓存
6. 销毁本地回放房间实例
7. 清空当前回放上下文
8. 恢复客户端在线事件处理能力
9. 将客户端状态机从 `InReplay` 安全迁回 `InLobby`

### 30.12 在线房间与回放房间的房间路由器实例隔离

在线房间与本地回放房间都可复用同一套 `ClientRoomMessageRouter` 类型结构，但不得复用同一实例。

强制约束如下：

1. 在线 `ClientRoomInstance` 必须持有自己的 `ClientRoomMessageRouter`
2. 本地回放房间对应的 `ClientRoomInstance` 必须持有自己的 `ClientRoomMessageRouter`
3. 二者只允许复用结构与处理机制，不允许复用实例、状态、订阅关系或上下文引用
4. `ReplayPlaybackController` 注入消息时，必须面向本地回放房间实例自身的 `ClientRoomMessageRouter`
5. 严禁将回放消息注入在线房间路由器实例
6. 严禁将在线房间延迟消息注入回放房间路由器实例
7. `ReplayPlaybackController` 在执行注入前，必须先校验：
   - 回放帧消息类型属于 `S2CRoomMessage`
   - 回放帧 `RoomId` 与当前本地回放房间一致
   - 当前本地回放房间处于可重演状态

## 31. 客户端回放观察者边界规范

### 31.1 回放客户端身份

客户端在回放模式下是上帝视角观察者。它是观看回放的存在，不是回放中的参与实体。

### 31.2 禁止控制回放重演实体

回放模式下，客户端不允许控制任何回放重演实体。

### 31.3 允许的控制能力

仅允许控制观察流程，例如：

- 播放
- 暂停
- 继续
- 快进
- 后退
- 跳转
- 倍速

以及观察类扩展功能，例如：

- 切换到某个录制客户端实体的观察视角
- 镜头跟随
- 时间点状态查看

### 31.4 观察视角切换不等于实体控制

即便支持切换到某个录制客户端所控制实体的视角，也只是观察视角切换，不代表获得该实体控制权。

## 32. 在线流程与回放流程隔离规范

### 32.1 输入源隔离

在线模式下，房间消息输入源来自服务端网络下行。回放模式下，房间消息输入源来自本地回放控制器按回放文件内容注入的模拟服务端事件流。

### 32.2 回放期间必须隔离房间域在线消息链

进入回放模式后，不要求强制断开真实网络连接。客户端可以继续保留全局域在线能力。但回放期间必须严格隔离房间域消息链。

### 32.3 当前激活房间上下文必须唯一

客户端在同一时刻，只允许存在一个当前激活的房间域上下文。该上下文要么是在线房间上下文，要么是回放房间上下文，不允许二者同时作为当前激活态存在。

## 33. EventBus 与 ServiceLocator 作用域规范

### 33.1 ServiceLocator 的作用域实现与接口边界

框架内置独立的 `ServiceLocator` 能力，但必须同时满足“底层复用”和“作用域语义隔离”两项要求。

约束如下：

1. `GlobalScope ServiceLocator` 与 `RoomScope ServiceLocator` 可共用一套底层泛型实现逻辑
2. 对外必须保持明确的作用域语义边界
3. 不允许将其退化为一个无作用域区分的大一统 Locator
4. `IGlobalService` 与 `IRoomService` 标记接口应保留，用于强化注册边界与作用域归属
5. 跨作用域误注册、误获取时，必须直接报错并阻断

### 33.2 ServiceLocator 使用边界

#### GlobalServiceLocator

只用于全局作用域内服务寻址。它不是状态仓库，不是跨域访问入口。

#### RoomServiceLocator

只用于单个房间作用域内服务寻址。它不能替代明确的业务契约设计。

### 33.3 EventBus 制度化约束

- `RoomEventBus` 与 `GlobalEventBus` 必须物理隔离
- EventBus 只负责域内解耦通知，不负责网络协议路由
- 高频热路径默认不建议走 EventBus
- 框架可提供轻量事件计数统计与 Warning 阈值

### 33.4 EventBus 的事件类型约束

EventBus 的事件类型必须具备明确的域标记约束，不允许采用纯泛型无约束模式。

约束如下：

1. `GlobalEventBus` 只允许处理实现了 `IGlobalEvent` 的事件类型
2. `RoomEventBus` 只允许处理实现了 `IRoomEvent` 的事件类型
3. 事件标记接口与协议基类不得混合承担双重语义
4. 任何普通 DTO、协议类型或无域归属类型，不得直接作为 EventBus 事件类型使用

### 33.5 EventBus 的默认派发模型

EventBus 默认必须采用**同步立即派发**模型。

约束如下：

1. 框架层不得依赖延迟派发才能正确工作
2. 事件发布后，默认应在当前调用链内完成派发
3. 若项目层自行扩展为缓冲或延迟派发，属于开发者自担时序风险的扩展行为
4. 框架默认不为项目层的延迟派发策略承担一致性兜底

### 33.6 EventBus.Clear() 的语义约束

`EventBus.Clear()` 的语义必须固定为：

1. 清空当前作用域内的全部订阅关系
2. 若实现存在待处理事件缓存，也必须一并清空
3. `Clear()` 调用后，该作用域内不得再保留任何旧订阅或旧事件残留

### 33.7 订阅绑定生命周期

- 全局业务单元订阅 `GlobalEventBus`，必须在其生命周期内注册，在退出前解除
- 房间业务单元订阅 `RoomEventBus`，必须在接入后生效，在销毁前解除

## 34. IRoomSettings 与配置系统规范

### 34.1 NetConfig 与房间配置边界

`NetConfig` 是服务端进程级配置。`IRoomSettings` 是单局房间业务配置快照。二者职责完全不同，不得混淆。

### 34.2 IRoomSettings 接口约束

`IRoomSettings` 是房间配置快照的**双端共同契约接口**，定义放在 **Shared 层**。它不等价于服务端进程配置文件模型，也不属于 `NetConfig.json` 这类服务端运行配置体系。

`IRoomSettings` 至少提供：

- `string SettingsFormat`
- `int SettingsVersion`
- `byte[] Serialize()`

约束如下：

1. `SettingsFormat` 用于标识当前房间配置快照格式
2. `SettingsVersion` 用于标识当前房间配置快照版本
3. `Serialize()` 仅返回业务配置快照字节内容
4. 框架可直接读取 `SettingsFormat` 与 `SettingsVersion` 写入回放文件头
5. 框架不解析 `Serialize()` 返回的业务字节内容，仅负责写入、读取与透传
6. `IRoomSettings` 属于房间配置快照契约，不属于服务端本地配置文件契约
7. `IRoomSettings` 的接口定义允许放在 Shared 层，其具体实现类必须按端归属，放在 Server 或 Client
8. 建房时，`RoomDispatcher` 必须先根据请求参数与服务端策略构建具体 `IRoomSettings`
9. `GlobalRoomManager.CreateRoom(...)` 必须接收该 `IRoomSettings`
10. `RoomInstance` 在构造阶段就必须持有有效 `IRoomSettings`，不得先创建不完整房间后再补挂配置

### 34.3 IRoomSettings 在服务端与客户端的职责划分

#### 服务端职责

- 作为权威房间配置承载对象
- 参与房间初始化与规则配置读取
- 在录像写入时提供 `SettingsFormat`、`SettingsVersion` 与 `Serialize()` 结果

#### 客户端职责

- 作为房间基础配置的本地只读载体
- 供房间 UI、View 或表现层读取基础房间规则信息
- 在回放模式下，根据回放文件头中的配置元数据与快照内容完成本地房间配置还原

### 34.4 NetConfig.json 配置与热重载策略

`NetConfig.json` 与 `NetConfigManager` 必须归属 **Server 端**，不属于 Shared 层公共契约，也不属于客户端共享配置体系。

- 静态配置：如 IP、端口，仅启动时读取，修改必须重启进程
- 动态配置：如大厅重连超时时长、房间空置超时时长、录像缓冲容量、Replay 下载超时、幂等缓存 TTL、幂等清理间隔等
- 热重载机制：框架提供 `NetConfigManager.Reload()`，仅刷新动态配置项；若检测到静态项变更，直接输出 Warning 并忽略

强制约束如下：

1. `NetConfigManager` 仅属于服务端运行时配置管理器
2. 客户端不得直接读取服务端 `NetConfig.json`
3. 客户端若需要感知部分运行参数，必须由服务端通过显式协议下发
4. 只有客户端运行逻辑确实需要消费的参数，才允许从服务端向客户端下发
5. 不允许将服务端内部运行配置整体镜像给客户端
6. 客户端若存在本地配置需求，应建立独立的客户端配置体系，而不是复用服务端网络配置文件
7. `NetConfigManager.Reload()` 必须由宿主层显式调用，框架不内置文件监听自动触发机制
8. 动态配置刷新后，默认只对后续新创建对象生效，不追溯改变已在运行中的房间实例

## 35. 幂等与防重规范

### 35.1 IdempotentCache 定位

`IdempotentCache` 只负责不可重复请求的防重。它不承担业务状态机职责。

### 35.2 建房幂等 Token 约束

`C2S_CreateRoom` 必须携带客户端本地生成的请求幂等 Token。服务端使用 `Dictionary<string, RoomCreateRecord>` 存储 Token 与处理结果。

约束如下：

1. 若处理前发现 Token 已存在且未过期，则直接返回原结果
2. Token 过期后若客户端以相同 Token 重试，视为新请求
3. 幂等缓存仅用于去重，不持有房间重量级业务对象引用
4. Token 必须为非空字符串
5. Token 校验失败时，服务端必须直接返回失败结果，不进入幂等查询逻辑

### 35.3 幂等结果记录边界

1. 允许缓存成功结果
2. 允许缓存明确失败结果
3. 所有结果统一受同一 TTL 管理
4. 同一 Token 在 TTL 内重复请求时，必须直接复用原始响应

### 35.4 Token 的客户端生成时机

1. 首次触发建房意图时生成 Token
2. 在收到成功或失败最终响应前，所有超时重试都必须复用同一 Token
3. 收到最终结果后结束该 Token 生命周期
4. 新一轮独立建房意图必须重新生成新 Token

### 35.5 幂等缓存的过期判定与清理配置约束

1. 命中时必须实时判定 TTL
2. 后台巡检只负责清理回收，不负责逻辑有效性
3. 若 `IdempotentCleanupIntervalSeconds >= IdempotentTtlSeconds`，框架必须输出 Warning

### 35.6 CreateRoom 请求期间的重复触发控制边界

框架不强制引入 `CreatingRoom` 这类中间主状态。等待 `S2C_CreateRoomResult` 期间，重复触发控制由 UI / 交互层自行处理，服务端再通过幂等 Token 最终兜底。

## 36. 房间空置超时与销毁兜底规范

### 36.1 房间空置超时规则

当房间内所有连接均处于断线状态，且持续时间超过配置值时，`GlobalRoomManager` 强制销毁该房间。

强制约束如下：

1. 房间因空置超时被销毁时，必须同步通知 Session 管理模块
2. Session 管理模块必须清空该房间成员会话上的 `CurrentRoomId`
3. Session 本身可继续保留到其独立超时结束
4. 后续重连时若会话仍有效但 `CurrentRoomId` 已为空，则服务端应返回 `S2C_ReconnectResult(Success)` 且目标状态为大厅
5. 客户端收到后进入 `InLobby`

### 36.2 Session 保留超时与 Room 空置超时分离

框架必须区分：

1. Session 保留超时
2. Room 空置销毁超时

二者必须独立配置、独立计时、独立生效。

## 40. 新人接入友好性规范

### 40.1 第一眼就能看懂

- 看协议名知道方向

- 看 `ExtData` 知道扩展承载位
- 看 `RoomId` 知道路由上下文
- 看日志知道错误位置

### 40.2 第一日接入目标

一个新开发者在接入框架第一天，应能快速完成以下事情，而不需要依赖口头教学：

- 新增协议
- 选择正确 `MessageId` 号段
- 理解协议只保留 `MessageId`
- 理解为什么禁止 `DeliveryMode` 协议标注
- 理解四协议基类自带 `ExtData` 扩展字段
- 理解四协议基类如何判断全局 / 房间与 C2S / S2C
- 在正确粒度下组织协议脚本
- 新增一个 Handle
- 新增一个 Model
- 新增一个 `RoomHandle`
- 新增一个 `RoomModel`
- 在正确目录下组织业务代码
- 理解出厂自带的全局模块目录和房间业务组件目录
- 理解房间基础设置组件是创建房间时强制挂载
- 理解其他房间业务组件是可选挂载并由开发者实现
- 理解 Handle 的统一注册入口应该写在哪里
- 理解链式 Register 风格应该如何书写
- 写全局域消息处理
- 写房间域消息处理
- 理解房间实例与房间业务单元的生命周期边界
- 理解房间路由隔离
- 理解断线重连职责边界
- 理解回放只能从大厅态进入
- 理解回放只能观察不能控制

### 40.3 任何高理解成本设计都应继续修订

如果某项设计需要开发者先理解大量隐式上下文、框架潜规则或历史遗留习惯，才能开始写业务代码，则说明该设计仍不够好，应继续收敛和标准化。

## 41. 启动期自检建议

建议在服务端与客户端启动阶段至少执行以下自检：

1. `MessageRegistry` 是否存在重复 `MessageId`
2. 所有协议类型是否都继承自四协议基类之一
3. 是否存在被错误注册到当前端入站链的反向协议类型
4. 所有房间域协议发送路径是否具备 `RoomId` 运行时绑定能力
5. Replay 录制旁路链是否只监听房间域 `S2C` 全体广播出口
6. 带有 `[MessageId]` 的协议是否全部符合最小元数据制度
7. EventBus 是否只接受对应域的事件类型
8. 全局与房间 ServiceLocator 是否存在跨域误注册
9. Adapter 发送入口是否只接收已完成上下文绑定的统一传输输入
10. 房间业务组件清单是否使用稳定注册标识而非运行时类型名
11. EventBus 默认是否为同步立即派发
12. Replay 记录结构是否以 Tick 为主轴而非直接复用线上原始封套
13. Shared 层是否未定义跨端统一发送器接口
14. 服务端发送链是否存在“目标集合解析完成后、进入 Adapter 前”的发送协调节点
15. `ServerRoomMessageRouter` 是否仅对当前已装配房间业务组件注册主处理 Handle
16. `ClientRoomMessageRouter` 是否仅对当前已装配客户端房间业务组件注册主处理 Handle
17. `NetConfigManager` 是否仅归属服务端，客户端未直接读取服务端配置文件
18. `MessageRegistry` 白名单程序集是否由各端启动装配器显式指定
19. `GlobalEventBus` 是否由 `GlobalInfrastructure` 持有
20. `GlobalRoomManager` 是否注册在 `GlobalInfrastructure` 的 `GlobalScope ServiceLocator` 中
21. `IRoomSettings` 是否作为 Shared 层双端共同契约接口存在，而未被误实现为服务端进程配置模型
22. `MessageRegistry` 是否通过显式构建器构建，而非静态自驱动初始化
23. 客户端房间域发送是否由调用方显式传入 `roomId`
24. `.sfr` 文件头是否显式写入必要版本与结构字段
25. 回放房间装配是否基于 `RoomComponentIds` 而非扫描录制帧反推
26. `ReplayPlaybackController` 是否由客户端全局门面直接持有
27. `RoomDispatcher` 与 `GlobalRoomManager` 是否已完成职责切割
28. 服务端顶级宿主是否通过 `GlobalInfrastructure` 统一驱动主循环
29. `ReplayRecorder` 是否由发送协调节点主动调用，而非订阅事件监听
30. 房间组件注册表是否由本端启动装配显式构建，而非运行期懒扫描
31. 进入回放前是否会先销毁当前在线 `ClientRoomInstance`
32. `RoomInstance` 是否区分成员身份集合与当前在线连接映射
33. 客户端状态协议过滤器是否位于 `ClientNetworkEntry` 与 Router 之间
34. 客户端房间装配器是否为实例级而非全局常驻单例
35. 房间组件任一实例化或初始化失败时，装配流程是否会立即整体阻断并执行回滚
36. `ConnectionId` 是否未被错误定义为 `NetworkEnvelope` 的共享传输字段
37. 房间归属一致性校验链是否只位于 `C2SRoomMessage` 路径上
38. `S2C_ReconnectResult` 的恢复装配是否由客户端全局门面统一协调
39. 回放消息是否直接注入本地回放房间的 `ClientRoomMessageRouter`
40. `C2S_Reconnect` 是否明确注册为 `C2SGlobalMessage`
41. 房间 Tick 是否明确为从 0 开始的相对 Tick

## 42. 统一禁止事项总表

禁止以下行为：

- 禁止 Infrastructure 直接展开业务 Handle 的消息监听细节
- 禁止匿名函数散落式注册协议处理
- 禁止不使用链式 Register 风格进行统一注册
- 禁止使用非 `C2S_XXX` / `S2C_XXX` 的协议命名
- 禁止开发者占用 `0 - 9999` 框架保留协议号段
- 禁止在业务协议上使用 `[DeliveryMode(DeliveryMode.ReliableOrdered)]` 标注
- 禁止在业务协议上继续叠加等价的传输模式标注
- 禁止在四协议基类之外重复表达方向与域归属语义
- 禁止共享契约层协议默认采用一协议一文件式过度拆分
- 禁止无收益地把局部小类型拆成大量细碎脚本
- 禁止把出厂自带模块和组件只做成空目录或空壳类
- 禁止出厂自带模块缺失协议、Handle、Model 等完整核心链路
- 禁止重新引入长期并存的 `GlobalModule + Handler` 双主概念
- 禁止重新引入长期并存的 `RoomComponent + Handle` 双主概念
- 禁止把 Sender 强行作为所有业务单元的固定层级
- 禁止同一业务边界的核心代码长期散落在多个无归属关系的目录
- 禁止在 `XXXRoomHandle` 外再包一层同职责的 `XXXRoomComponent` 作为业务主入口
- 禁止把 `RoomInstance` 当前 `Running` 直接当成业务正式开始
- 禁止等待开始阶段录制正式对局事件
- 禁止把广播是否发生等价成是否允许录制
- 禁止客户端自行裁定房间真实状态
- 禁止跳过房间基础设置组件的强制挂载
- 禁止把房间基础设置组件做成可选挂载
- 禁止在房间运行中随意插拔未声明支持热接入的房间业务组件
- 禁止把同一个 Handle 的监听清单拆散到多个无边界位置
- 禁止房间业务单元绕开 `RoomInstance` 自行维护主生命周期
- 禁止非大厅态进入回放
- 禁止在线房间上下文与回放房间上下文同时作为当前激活态存在
- 禁止在回放期间发起进入房间或创建房间
- 禁止在在线房间中直接切入回放
- 禁止回放模式控制任何重演实体
- 禁止把观察视角切换实现为实体控制权切换
- 禁止在线流程和回放流程混用
- 禁止跨房间串路由
- 禁止第一个房间收到第二个房间的消息
- 禁止用低信息量方式生成正式 `RoomId`
- 禁止新增房间业务单元不支持断线重连就接入框架
- 禁止 View、UI、动画、音效、特效直接监听网络协议
- 禁止 EventBus 直接承担原始协议广播职责
- 禁止房间业务组件自行直接注册 / 注销 Router
- 禁止 Shared 层承载发送器实现
- 禁止在常规业务逻辑中使用 try-catch 掩盖错误
- 禁止依赖客户端隐式当前房间上下文自动补全房间域发送 `RoomId`

## 43. 实施顺序建议

### 第一阶段：统一协议命名、元数据、号段、目录、脚本粒度与注册风格

优先收口：

- 协议统一命名为 `C2S_XXX` / `S2C_XXX`
- 四协议基类统一决定上下行与域归属
- 协议只保留 `MessageId`
- 明确禁止 `DeliveryMode` 协议标注
- 四协议基类统一预留 `ExtData`
- 协议 ID 号段统一执行
- 协议脚本按业务边界聚合
- 全局业务单元统一为 Handle-Model
- 房间业务单元统一为 RoomHandle-Model
- 全局业务代码按 `XXXGlobalModule` 目录集中收口
- 房间业务代码按 `XXXRoomComponent` 目录集中收口
- 明确房间基础设置组件为创建房间时强制挂载项
- 全局业务 Handle 改为链式 `RegisterAll`
- 房间业务 Handle 改为集中注册
- Registrar 抽象统一接入风格

### 第二阶段：补全出厂自带模块和组件骨架内容

收口：

- 每个出厂模块补齐协议聚合脚本
- 每个出厂模块补齐 Handle / Model
- 每个出厂组件补齐协议聚合脚本
- 每个出厂组件补齐 RoomHandle / RoomModel
- 默认常用功能范围落地

### 第三阶段：房间生命周期与录制边界改造

收口：

- `RoomInstance` 作为唯一生命周期源
- `RoomInstance` 运行态与房间业务态分离
- 房间业务单元只响应房间实例阶段回调
- 创建房间时强制挂载房间基础设置组件
- 其他房间组件采用开发者可选挂载策略
- 录制只在游戏中阶段生效
- 录制内容以服务端权威事件结果为主

### 第四阶段：房间路由与重连恢复改造

收口：

- 每房间独立 `RoomId` 路由
- 严格串房阻断
- 业务单元清单驱动重建
- 业务单元快照恢复
- RoomHandle / RoomModel 恢复职责边界

### 第五阶段：回放观察者边界与入口态改造

收口：

- 回放客户端是上帝视角观察者
- 回放只能从大厅态进入
- 进入房间与进入回放互斥
- 回放由本地控制器模拟服务端事件重演
- 禁止控制重演实体
- 仅允许观察流程控制与观察类扩展

## 45. 结语

本次合并修订的核心目标，不是增加更多抽象层，而是把当前框架中最关键、最容易出错、最影响开发者体验的制度彻底写清楚并固定下来。

真正必须统一的核心事项包括：

- 协议怎么命名
- 协议如何通过四协议基类判断所属域与上下行
- 协议只保留什么元数据
- 为什么协议只保留 `MessageId`
- 为什么禁止 `DeliveryMode` 一类协议标注
- 四协议基类为什么统一预留 `ExtData`
- 协议 ID 用哪个号段
- 协议脚本应该按什么粒度组织
- 业务单元到底采用什么结构
- 业务代码目录应该怎么收口
- 出厂自带哪些全局模块和房间业务组件
- 出厂自带模块与组件必须补齐哪些骨架内容
- 房间基础设置组件为什么必须强制挂载
- 房间实例为什么必须作为房间域唯一生命周期源
- 房间业务组件为什么只能响应房间实例调用
- 房间路由如何隔离
- 录制什么时候开始
- 重连恢复由谁负责
- 回放如何进入
- 回放和在线房间如何隔离
- 回放客户端到底能做什么、不能做什么

只要这些关键制度被写死、被统一、被评审门禁化，开发者就不需要再浪费时间猜框架。框架的价值，正是让开发者把第一天、第二天以及后续迭代中的主要精力，都投入到游戏业务逻辑本身，而不是投入到理解框架历史习惯和隐式规则之中。
